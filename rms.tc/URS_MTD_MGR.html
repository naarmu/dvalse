URS_MTD_MGR

Method 조회 · 관리

User Requirement Core Function
01

기능 개요

서버에 등록된 모든 Method를 한 화면에서 조회·검색·관리하는 RMS의 Method 진입 화면이다. 사용자는 이 화면을 통해 어떤 Method가 어떤 버전으로 존재하는지 파악하고, 상세 파라미터를 확인하며, 다음 행위(비교·배포·삭제·로컬 파일 입출력)를 시작한다.

Recipe Manager와 동일한 운영 모델을 따르되, Method 영역에서는 별도의 버전 관리 화면이나 Direct 업·다운로드 화면이 없으며 이 모든 기능이 Method Manager에 통합되어 있다.

02

사용자 니즈

주요 사용자

  • 공정 엔지니어 — 특정 Method의 현재 파라미터를 빠르게 확인
  • 장비 엔지니어 — 어떤 장비에 어떤 Method가 있는지 추적
  • 공정 표준화 팀 — 전체 Method 자산을 한눈에 조감

Pain Point (RMS 도입 전)

  • 장비별로 흩어진 Method를 일일이 들여다봐야 함
  • 같은 이름의 Method라도 장비마다 내용이 다른 경우가 흔함
  • Method 변경 이력이 추적되지 않아 사후 검토가 어려움

기대 효과

  • 모든 Method가 단일 화면에서 트리·목록 형태로 조회
  • 버전별 변경 시각·변경자를 통한 이력 가시화
  • 로컬 파일 직접 입출력으로 외부 자료와의 교환 용이
03

기능 요구사항

■ FR-1. Method 목록 조회

  • FR-1.1 화면 진입 시 서버에 등록된 모든 Method가 좌측 트리 형태로 표시되어야 한다.
  • FR-1.2 각 Method 노드 하위에는 보유한 모든 버전이 펼침/접힘 가능한 형태로 표시되어야 한다.
  • FR-1.3 각 버전에는 버전 번호와 수집/등록 일시, 변경자가 함께 표시되어야 한다.
  • FR-1.4 Method 출처(Collected / Direct Upload)를 시각적으로 구분 표시한다.

■ FR-2. 상세 정보 표시

  • FR-2.1 Method 또는 버전 선택 시 중앙 영역에 Method Information(타입, 버전, 최종 수정일, 수집 장비, 설명)이 표시되어야 한다.
  • FR-2.2 우측 영역에 Method Contents(파라미터 목록)가 표시되어야 한다.
  • FR-2.3 다른 버전으로 전환 시 두 영역이 즉시 갱신되어야 한다.
  • FR-2.4 표시되는 파라미터는 Manage Method Parameters의 표시/숨김 설정에 따른다.

■ FR-3. 검색 · 정렬 · 필터

  • FR-3.1 사용자는 Method 이름으로 실시간 검색할 수 있어야 한다. 대소문자를 구분하지 않으며, 부분 일치를 지원한다.
  • FR-3.2 사용자는 Method 목록을 이름 또는 수집일 기준으로 정렬할 수 있어야 한다.
  • FR-3.3 정렬 기본값은 수집일 내림차순이며, 가장 최근 변경된 Method가 상단에 위치한다.
  • FR-3.4 사용자는 특정 장비를 선택하여 해당 장비에서 수집된 Method만 필터링할 수 있어야 한다.

■ FR-4. Direct Upload (로컬 → 서버)

  • FR-4.1 사용자는 로컬 PC의 Method 파일을 파일 선택 또는 드래그 앤 드롭으로 업로드할 수 있어야 한다.
  • FR-4.2 업로드된 Method는 트리에 신규 Method로 표시되며, 출처는 "Direct Upload"로 시각 구분된다.
  • FR-4.3 동일 이름 Method가 이미 존재하는 경우, 새 버전 추가 / 덮어쓰기 / 취소 옵션을 제공한다. 기본값은 새 버전 추가이다.
  • FR-4.4 업로드 파일은 확장자·매직 바이트·안티바이러스 검증을 거쳐야 등록된다.

■ FR-5. Direct Download (서버 → 로컬)

  • FR-5.1 사용자는 선택한 Method를 로컬 PC로 다운로드할 수 있어야 한다.
  • FR-5.2 다운로드 파일명은 자동 규칙에 따라 생성된다: <Method명>_v<버전>_<날짜>.
  • FR-5.3 다운로드 이력은 Method History에 자동 기록된다.

■ FR-6. Method 삭제

  • FR-6.1 사용자는 Method 또는 특정 버전을 선택하여 삭제할 수 있어야 한다.
  • FR-6.2 삭제 전 확인 팝업에는 Method 이름, 보유 버전 수, 영향 안내가 명시되어야 한다.
  • FR-6.3 현재 어떤 장비에 배포 중인 Method를 삭제할 경우, 배포된 장비 목록을 명시한 경고를 표시한 후 사용자 확인을 받는다.
  • FR-6.4 삭제는 Soft Delete로 처리되며, 30일간 복원 가능해야 한다.
  • FR-6.5 삭제 이력은 Method History에 자동 기록된다.

■ FR-7. UI 상태 표시

  • FR-7.1 데이터 로딩 중에는 로딩 인디케이터를 300ms 이후 표시한다.
  • FR-7.2 등록된 Method가 없는 경우 Empty State 안내 메시지와 다음 액션 유도(Direct Upload 바로가기 등)를 제공한다.
04

비기능 요구사항

■ NFR-1. 성능

  • NFR-1.1 일반 환경에서 화면 진입 후 5초 이내에 모든 Method 목록이 표시되어야 한다.
  • NFR-1.2 Method 선택 후 상세 정보 표시까지 1.5초 이내.
  • NFR-1.3 검색 입력 시 응답 지연은 300ms 이내.

■ NFR-2. 권한

  • NFR-2.1 모든 사용자(All Roles)는 Method 조회·검색·정렬·필터링이 가능해야 한다.
  • NFR-2.2 Direct Upload는 Vendor, Admin, Engineer만 수행 가능하다. Direct Download는 모든 사용자에게 허용한다.
  • NFR-2.3 Method 삭제는 Vendor, Admin만 수행 가능하다.
  • NFR-2.4 권한 검증은 UI 제약과 서버 API 이중 방어로 수행되어야 한다.

■ NFR-3. 데이터 무결성

  • NFR-3.1 업로드·다운로드 이력에 파일 해시값(SHA-256)을 포함하여 사후 검증이 가능해야 한다.
  • NFR-3.2 다운로드 → 재업로드 시 파일 내용이 동일해야 한다(라운드트립 무손실).
05

제약사항 · 정책

  • 정책 Method 영역은 Recipe와 달리 버전 관리·Direct 업·다운로드 별도 화면을 두지 않고 Method Manager에 통합한다. Method가 Recipe보다 단순한 자료 구조이며 운영 빈도도 낮은 점을 반영한 결정이다.
  • 정책 Direct Upload된 Method는 Direct Upload 출처 아이콘으로 시각 구분된다.
  • 정책 동명 Method 업로드 시 기본 동작은 새 버전 추가이다.
  • 정책 Method 삭제는 Soft Delete 방식으로 처리하며 30일간 복원 가능하다.
  • 정책 정렬 기본값은 수집일 내림차순, 버전 정렬은 최신 버전 상단이다.
  • 제약 Direct Upload 권한은 Vendor, Admin, Engineer로 한정된다. Direct Download는 모든 사용자에게 허용된다.
  • 제약 Method 삭제 권한은 Vendor, Admin로 한정된다.
  • 제약 업로드 파일은 안티바이러스 스캔을 통과해야 서버에 등록된다.
