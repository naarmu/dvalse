URS_CLT_RCP

Recipe 자동 수집

User Requirement Core Function
01

기능 개요

장비 PC에 설치된 Agent가 장비의 Recipe 파일 변경을 감지하여 RMS 서버로 자동 전송하는 백그라운드 기능이다. 사용자가 직접 조작하는 화면은 아니지만, RMS 자산이 채워지는 가장 기본적인 경로이다.

Agent는 장비의 Recipe 디렉토리를 모니터링하다가 새 파일이 생기거나 기존 파일이 변경되면 이를 서버로 전송한다. 전송된 Recipe는 즉시 자산에 편입되지 않고 Pending Recipes를 거쳐 사용자 검토 후 등록된다.

02

사용자 니즈

주요 사용자

  • 장비 엔지니어 — 장비에서 만든 Recipe가 자동으로 서버에 모이는 것을 기대
  • RMS 관리자 — 수집 상태 점검

Pain Point (RMS 도입 전)

  • 장비에서 만든 Recipe를 별도 절차로 서버에 올려야 함
  • 수집 누락이 발생해도 사후에 인지하기 어려움
  • 장비 오프라인 시점의 변경 사항이 영구히 사라질 위험

기대 효과

  • 사용자 개입 없이 모든 Recipe 변경이 서버에 자동 동기화
  • 오프라인 장비도 재연결 시 누락 없이 수집
  • 수집 실패에 대한 자동 재시도와 가시화
03

기능 요구사항

■ FR-1. 변경 감지

  • FR-1.1 Agent는 장비의 Recipe 디렉토리를 모니터링하여 파일 변경(생성 / 수정 / 삭제)을 감지해야 한다.
  • FR-1.2 감지 방식은 파일 시스템 이벤트 기반 또는 주기적 폴링 중 하나를 사용한다.
  • FR-1.3 감지 즉시 또는 일정 디바운스 후 서버 전송 작업이 수행된다.

■ FR-2. 서버 수신 · Pending 등록

  • FR-2.1 서버는 Agent가 전송한 Recipe를 수신하여 Pending Recipes 목록에 추가한다.
  • FR-2.2 Pending 항목에는 장비명, Recipe명, 변경 유형(신규 / 변경 / 삭제 추정), 수집 시각이 포함된다.
  • FR-2.3 Pending 처리는 별도 화면(URS_SYS_PND)에서 사용자가 검토하여 결정한다.

■ FR-3. 수집 주기 · 트리거 설정

  • FR-3.1 Vendor·Admin은 Agent의 수집 주기 또는 트리거 방식을 설정할 수 있어야 한다.
  • FR-3.2 설정 변경은 해당 Agent에 반영되며, 다음 수집부터 적용된다.
  • FR-3.3 설정 변경 이력은 자동 기록된다.

■ FR-4. 오프라인 처리

  • FR-4.1 장비가 서버와 연결되지 않은 상태에서 발생한 Recipe 변경은 Agent가 로컬 큐에 보존한다.
  • FR-4.2 서버 재연결 시 큐에 보존된 항목들이 순차적으로 전송된다.
  • FR-4.3 큐에 보관된 항목 수에 임계가 있어, 임계 초과 시 사용자 알림이 발생한다.

■ FR-5. 반복 변경 처리

  • FR-5.1 동일 Recipe가 반복 변경되어 수집된 경우, 각 변경이 별도 버전으로 관리된다.
  • FR-5.2 사용자가 Pending에서 등록 시점에 어떤 버전이 등록되는지 명확히 표시된다.
  • FR-5.3 짧은 시간 내 동일 파일의 다중 변경은 Agent가 디바운스 처리하여 마지막 상태만 전송할 수 있다.

■ FR-6. 수집 실패 처리

  • FR-6.1 수집 실패(네트워크 오류, 파일 검증 실패 등) 시 Agent는 일정 횟수 자동 재시도한다.
  • FR-6.2 재시도가 모두 실패한 경우 에러 로그를 서버에 보고한다.
  • FR-6.3 수집 실패는 RMS Log 또는 Server Log에 기록되며, 관리자가 실패 패턴을 추적할 수 있어야 한다.
04

비기능 요구사항

■ NFR-1. 데이터 무결성

  • NFR-1.1 Agent는 파일을 완전히 읽기 전까지 전송을 시작하지 않는다(파일 잠금 또는 안정성 검증 후 전송).
  • NFR-1.2 전송 중 끊김이 발생해도 부분 파일이 서버에 등록되지 않는다.
  • NFR-1.3 전송 시 SHA-256 해시를 함께 보내 서버에서 무결성 검증이 가능하다.

■ NFR-2. 성능 · 부하

  • NFR-2.1 Agent의 모니터링 동작은 장비 본래 작업에 영향을 주지 않을 만큼 가벼워야 한다.
  • NFR-2.2 다수 장비가 동시 수집 중인 상황에서도 서버가 적절히 분산 처리해야 한다.

■ NFR-3. 보안

  • NFR-3.1 Agent–서버 통신은 TLS 암호화 채널 위에서 수행된다.
  • NFR-3.2 수집된 파일은 서버 등록 전 안티바이러스 스캔을 통과해야 한다.

■ NFR-4. 가시성

  • NFR-4.1 수집 상태(성공·실패·대기)는 Equipment Dashboard 또는 Connection 화면에서 확인할 수 있어야 한다.
05

제약사항 · 정책

  • 정책 수집된 Recipe는 자동으로 자산에 편입되지 않는다. Pending Recipes를 거쳐 사용자 검토 후 등록된다.
  • 정책 오프라인 장비의 변경 사항은 Agent의 로컬 큐에 보존되며, 재연결 시 자동 전송된다.
  • 정책 동일 파일의 짧은 시간 내 다중 변경은 디바운스되어 마지막 상태만 전송될 수 있다.
  • 정책 수집 설정 변경(주기·트리거)은 자동 기록되며 수정·삭제할 수 없다.
  • 제약 Agent–서버 통신은 TLS 암호화 채널 위에서만 허용된다.
  • 제약 모든 전송에 SHA-256 해시가 동반되어야 한다(무결성 검증).
  • 제약 수집 설정 변경 권한은 Vendor, Admin로 한정된다.
06

관련 Test Cases

이 URS의 요구사항을 검증하는 QA 테스트 케이스 6건:

  • TC_CLT_RCP_001 — 장비측 Recipe 변경 감지 및 서버 수신 (FR-1, FR-2)
  • TC_CLT_RCP_002 — 수집된 Recipe가 Pending Recipes에 표시 (FR-2)
  • TC_CLT_RCP_003 — 수집 주기 · 트리거 설정 동작 (FR-3)
  • TC_CLT_RCP_004 — 장비 오프라인 상태에서 수집 재개 (FR-4)
  • TC_CLT_RCP_005 — 동일 Recipe 반복 변경 시 버전 관리 (FR-5)
  • TC_CLT_RCP_006 — 수집 실패 시 재시도 및 에러 로그 (FR-6)
