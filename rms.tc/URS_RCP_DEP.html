URS_RCP_DEP

Recipe 배포

User Requirement Core Function
01

기능 개요

서버에 등록된 Recipe의 특정 버전을 하나 또는 다수의 장비에 전송·반영하는 기능이다. RMS의 가장 핵심적인 "출력" 행위이며, 공정에 실제 영향을 주는 유일한 경로이다.

Recipe Manager가 읽기·관리의 진입점이라면, Deploy Recipes는 실행·반영의 진입점이다. 사용자는 이 화면을 통해 공정 표준(Master Recipe)을 장비 현장까지 물리적으로 전달한다.

02

사용자 니즈

주요 사용자

  • 공정 엔지니어 — 공정 조건 변경 시 여러 장비에 동일 Recipe를 빠르게 반영해야 함
  • 장비 엔지니어 — 장비 유지보수 후 표준 Recipe를 복구 배포
  • 공정 표준화 팀 — 새 Recipe Master를 여러 FAB에 전파

Pain Point (RMS 도입 전)

  • 각 장비에 USB·공유 폴더로 Recipe를 수동 복사 → 실수·누락·버전 혼선이 반복됨
  • "이 장비에 지금 어떤 Recipe가 깔려 있지?"가 한눈에 안 보임
  • 배포 이력이 개인 메모·메신저 기록에만 남아 감사 대응 곤란

기대 효과

  • 다수 장비에 일괄 배포 → 표준화 작업 시간 단축
  • Master 버전 중심의 일관된 공정 운영
  • 모든 배포 이력의 자동 기록 → SEMI 감사 대응 즉시 가능
03

기능 요구사항

■ FR-1. 배포 대상 선정

  • FR-1.1 사용자는 서버에 등록된 Recipe 중 하나를 선택할 수 있어야 한다.
  • FR-1.2 선택한 Recipe의 Available Versions 목록이 표시되며, 사용자는 배포할 버전을 지정할 수 있어야 한다.
  • FR-1.3 Master로 지정된 버전은 목록에서 시각적으로 구분되어야 하며, 초기 기본 선택값은 Master여야 한다.

■ FR-2. 배포 대상 장비 선정

  • FR-2.1 사용자는 개별 장비를 하나 선택하여 단일 배포할 수 있어야 한다.
  • FR-2.2 사용자는 여러 장비를 동시에 선택하여 다중 배포할 수 있어야 한다.
  • FR-2.3 사용자는 장비 그룹을 선택하여 소속 장비 전체에 일괄 배포할 수 있어야 한다.
  • FR-2.4 Offline 상태 장비는 배포 대상에서 기본 제외되며, 사용자에게 명확히 표시되어야 한다.

■ FR-3. 배포 실행 · 확인

  • FR-3.1 배포 실행 전 확인 팝업에는 대상 장비 목록, Recipe 이름, 버전이 명시되어야 한다.
  • FR-3.2 확인 팝업에는 각 장비의 현재 배포된 Recipe → 새 Recipe 전환 내역이 표시되어야 한다. (실수 배포 예방)
  • FR-3.3 사용자는 확인 단계에서 취소할 수 있어야 하며, 취소 시 아무런 변경이 발생하지 않아야 한다.

■ FR-4. 배포 진행 상태 표시

  • FR-4.1 배포 시작 즉시 진행 UI가 표시되어야 한다.
  • FR-4.2 다중 장비 배포 시 장비별 상태(Pending / In Progress / Success / Failed)가 개별 표시되어야 한다.
  • FR-4.3 사용자는 배포 진행 중에도 다른 화면으로 이동할 수 있어야 하며, 상단 알림 영역에서 진행률을 확인할 수 있어야 한다.

■ FR-5. 배포 결과 · 재시도

  • FR-5.1 배포 완료 후 종합 결과(성공 N / 실패 M)가 표시되어야 한다.
  • FR-5.2 실패한 장비의 실패 원인이 기술적 상세 + 사용자 친화 설명 + 해결 힌트의 세 층위로 제공되어야 한다.
  • FR-5.3 사용자는 실패한 장비만 선택하여 원클릭으로 재시도할 수 있어야 한다.
  • FR-5.4 재시도 이력은 원 실패 이력과 별도 레코드로 기록되어야 한다. (원 이력은 append-only로 보존)

■ FR-6. 이력 기록

  • FR-6.1 모든 배포는 Deployment Log에 자동 기록되어야 한다. 필수 필드는 배포자 계정, 일시, Recipe 이름·버전, 대상 장비, 결과, 실패 시 원인이다.
  • FR-6.2 다수 장비 배포 시 장비별로 각각 레코드가 생성되어야 한다. (장비별 독립 추적 가능)
  • FR-6.3 Deployment Log는 수정·삭제가 불가능(append-only)해야 한다.
04

비기능 요구사항

■ NFR-1. 데이터 무결성

  • NFR-1.1 Agent는 Recipe 파일 수신 시 임시 파일로 먼저 기록 → 완전 수신 후 최종 rename하는 atomic write 방식을 적용해야 한다. 전송 중 끊김이 있어도 장비의 기존 Recipe가 손상되지 않아야 한다.
  • NFR-1.2 다수 장비 배포 시 하나의 장비 실패가 다른 장비의 배포를 중단시키지 않아야 한다.

■ NFR-2. 성능

  • NFR-2.1 단일 장비 배포는 평균 10초 이내에 완료되어야 한다. (네트워크 정상 조건)
  • NFR-2.2 동시 배포 최대 장비 수는 10대로 제한한다. 초과 시 순차 배포로 자동 전환한다.

■ NFR-3. 권한

  • NFR-3.1 배포는 Engineer 이상 권한 계정만 수행할 수 있어야 한다. (Operator 차단)
  • NFR-3.2 권한 검증은 UI 제약과 서버 API 이중 방어로 수행되어야 한다.

■ NFR-4. 감사 대응

  • NFR-4.1 Deployment Log의 보존 기간은 최소 3년이다. (반도체 업계 감사 요구사항 기준)
  • NFR-4.2 Deployment Log는 SEMI 감사 Export 포맷으로 다운로드 가능해야 한다. (CSV + 해시값 + 생성자 메타)
05

제약사항 · 정책

  • 정책 한 Recipe의 Master 버전은 항상 1개이며, Deploy Recipes의 기본 선택값으로 사용된다.
  • 정책 Offline 장비에 대한 배포는 배포 대기열에 등록 후 재연결 시 자동 시도한다. 자동 배포 전 별도 관리자 승인은 요구하지 않는다.
  • 정책 현재 배포 중인 Recipe를 삭제할 경우, 경고 팝업 + 배포된 장비 목록 명시 후 삭제를 허용한다.
  • 제약 동시 배포 장비 수는 최대 10대. 초과분은 순차 처리.
  • 제약 배포는 Engineer, Admin, Vendor만 수행 가능. Operator는 조회만 허용.
  • 제약 Deployment Log는 어떤 사용자도 수정·삭제할 수 없다. (append-only)

  • TC_RCP_DEP_003 — 다수 장비 동시 배포 (4대 이상) (FR-2.2, NFR-2.2)
  • TC_RCP_DEP_004 — 장비 그룹 단위 배포 (FR-2.3, FR-2.4)
  • TC_RCP_DEP_005 — 배포 중 장비 연결 끊김 시 에러 처리 (NFR-1.1, FR-5.2)
  • TC_RCP_DEP_006 — 배포 완료 후 Deployment Log 기록 (FR-6)
  • TC_RCP_DEP_007 — 배포 진행률 · 상태 표시 (FR-4)
  • TC_RCP_DEP_008 — 배포 결과 장비별 성공 · 실패 구분 표시 (FR-5.1, FR-5.3)
