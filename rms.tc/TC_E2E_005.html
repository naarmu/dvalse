TC_E2E_005

계정 생성(Engineer Role) → Privilege에서 Deploy 권한 제거 → 해당 계정 로그인 시 접근 제한 검증

Permission Critical
01

Precondition

  1. Vendor 계정 로그인
  2. 기본 Engineer Role에는 Deploy 권한이 활성화된 상태
02

Test Steps

  1. Vendor가 Account 관리에서 신규 Engineer 계정 생성 (TC_SET_ACC_001 연계)
  2. 생성한 계정의 ID/PW 확인
  3. Privilege에서 Engineer Role의 Deploy 권한 체크 해제 (TC_SET_PRV_002 연계)
  4. 저장 후 변경 이력 기록 확인 (TC_SET_PRV_009 연계)
  5. Vendor 로그아웃 후 생성한 Engineer 계정으로 로그인 (TC_SYS_LGN_001)
  6. 최초 로그인 시 PW 강제 변경 확인 (TC_SET_ACC_005 연계)
  7. Deploy Recipes 메뉴 접근 시도 → 차단 확인 (TC_SET_PRV_007 연계)
  8. Recipe Manager, Compare Recipes 등 조회 기능은 정상 동작 확인
  9. 차단 이력이 RMS Log에 기록되는지 확인 (TC_HST_RMS_002 연계)
03

Test Data

New Account: engineer_test_001 / Role: Engineer Removed Privilege: Deploy Expected: 로그인 가능, 조회 가능, 배포 차단
04

Expected Result

  1. 계정 생성이 정상 진행되고 초기 PW 강제 변경 동작
  2. Privilege 변경이 즉시 저장, 이력 기록
  3. 새 Engineer 계정은 조회만 가능하고 Deploy 관련 메뉴·버튼이 숨김 또는 비활성
  4. API 레벨에서도 Deploy 요청이 차단 (서버 측 검증)
  5. 권한 거부 시도가 RMS Log에 기록
05

비고 (QA · 개발 메모)

보안 핵심: 권한 설정과 실제 적용이 일치하는지 검증하는 핵심 테스트. 서버 측 검증이 반드시 동작해야 함.

06

제안 사항

보안 Privilege 설정은 UI 제약 + API 서버 측 검증 이중 방어가 필수입니다. UI만으로는 브라우저 개발자 도구 등으로 우회 가능합니다.

데이터 무결성 권한 거부로 차단된 시도(403 에러 등)도 RMS Log에 기록해야 이상 행위 탐지가 가능합니다.
