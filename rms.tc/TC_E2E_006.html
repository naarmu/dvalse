TC_E2E_006

Method 수집 → 버전 비교 → 다수 장비 배포 → Deployment Log 상세 조회

Functional High
01

Precondition

  1. Online 장비 3대 이상 (EQ_001, EQ_002, EQ_003)
  2. EQ_001에 기존 Method X가 존재
02

Test Steps

  1. EQ_001의 Method X를 장비에서 일부 파라미터 변경 (파일 편집)
  2. Agent가 변경 감지하여 Pending에 등록 (TC_CLT_MTD_001 연계)
  3. Pending에서 v2로 서버 등록
  4. Method Manager에서 Method X의 v1, v2 확인 (TC_MTD_MGR_006 연계)
  5. Compare Methods로 v1, v2 비교 (TC_MTD_CMP_004 연계)
  6. Manage Method Versions에서 v2를 Master로 지정 (TC_MTD_VER_002 연계)
  7. Deploy Methods에서 EQ_001, EQ_002, EQ_003 세 장비 선택 후 Master 배포 (TC_MTD_DEP_003 연계)
  8. 배포 완료 대기
  9. Deployment Log로 이동, Type=Method 필터 적용 (TC_HST_DEP_001 연계)
  10. 방금 수행한 배포 3건의 상세 정보 확인 (배포자, 시각, Method명, 버전, 장비)
03

Test Data

Method X: v1 → v2 (Master 지정) → 3대 장비 배포 End: Deployment Log 3건 기록
04

Expected Result

  1. Method의 수집·비교·Master 지정·다중 배포가 Recipe와 동일한 UX로 동작
  2. Deployment Log에서 Method 배포가 Recipe와 구분되어 필터링됨
  3. 3건 모두 성공 기록
  4. Method History에서도 시계열 추적 가능
05

비고 (QA · 개발 메모)

Recipe와의 일관성: Method 관련 플로우가 Recipe와 동일하게 동작해야 사용자 학습 비용이 낮음.

06

제안 사항

일관성 Recipe와 Method의 운영 UX가 구조적으로 동일하도록 설계하는 것이 중요합니다. 현재 testcases.js에서 RCP 54개 + MTD 31개 구조인데, 차이가 나는 부분(RCP는 Custom XML 있음, MTD는 없음 등)을 명확히 문서화해주세요.
