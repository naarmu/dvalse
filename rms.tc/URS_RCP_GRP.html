URS_RCP_GRP

Recipe 그룹 관리

User Requirement Organization
01

기능 개요

여러 Recipe를 의미 있는 단위로 묶어 관리하는 기능이다. 사용자는 Recipe를 공정별, 용도별, 장비별 등 다양한 관점으로 분류하여 그룹을 만들고, 그룹 단위로 조회하거나 일괄 작업을 수행한다.

Recipe 수가 늘어남에 따라 단순한 평면 목록만으로는 관리가 어려워진다. Recipe 그룹은 운영 맥락에 맞는 분류 체계를 사용자가 직접 정의할 수 있도록 하는 기능이다.

02

사용자 니즈

주요 사용자

  • 공정 표준화 팀 — 공정별, 용도별 Recipe 분류
  • RMS 관리자 — Recipe 자산 관리 체계 구성
  • 공정 엔지니어 — 자주 사용하는 Recipe 묶음 빠른 접근

Pain Point (RMS 도입 전)

  • Recipe가 많아질수록 원하는 항목을 찾기 어려움
  • 같은 용도의 Recipe들을 묶어서 다루고 싶지만 평면 목록만으로는 한계
  • 관점이 여럿(공정·장비·용도)인데 단일 분류만으로는 부족

기대 효과

  • Recipe 자산을 운영 맥락에 맞춰 분류 가능
  • 그룹 단위 조회·일괄 작업으로 작업 효율 향상
  • 다중 그룹 소속을 지원하여 복수 관점의 분류 동시 적용 가능
03

기능 요구사항

■ FR-1. 그룹 생성

  • FR-1.1 사용자는 새 Recipe 그룹을 생성할 수 있어야 한다.
  • FR-1.2 그룹 생성 시 Group Name과 초기 멤버 Recipe를 선택할 수 있다.
  • FR-1.3 동일한 이름의 그룹은 생성할 수 없다. 중복 시 명확한 에러 메시지가 표시된다.
  • FR-1.4 그룹명 중복 검사는 대소문자를 구분하지 않는다.

■ FR-2. 멤버 편집

  • FR-2.1 사용자는 기존 그룹에 Recipe를 추가하거나 제거할 수 있어야 한다.
  • FR-2.2 드래그 앤 드롭 방식으로 Recipe 이동을 지원한다.
  • FR-2.3 멤버 변경 사항은 다른 화면(Recipe Manager 등)에 즉시 반영된다.
  • FR-2.4 한 Recipe는 여러 그룹에 동시 소속될 수 있다.

■ FR-3. 그룹 정보 변경

  • FR-3.1 사용자는 그룹 이름을 변경할 수 있어야 한다.
  • FR-3.2 이름 변경 시 변경 이력이 기록되며, 이전 이름이 함께 보존된다.
  • FR-3.3 이름 변경은 다른 화면에 즉시 반영된다.

■ FR-4. 그룹 삭제

  • FR-4.1 사용자는 그룹을 삭제할 수 있어야 한다.
  • FR-4.2 소속 Recipe가 있는 그룹을 삭제할 때, 사용자에게 경고가 표시된다.
  • FR-4.3 그룹 삭제 시 그룹만 제거되며, 소속 Recipe 자체는 유지된다. Recipe는 "미지정" 상태로 복귀한다.
  • FR-4.4 단, 해당 Recipe가 다른 그룹에도 소속된 경우, 그 그룹과의 관계는 유지된다.

■ FR-5. 다른 화면과의 연동

  • FR-5.1 그룹 정보는 Recipe Manager의 트리에 그룹 노드로 표시된다.
  • FR-5.2 Deploy Recipes 등의 화면에서 Recipe 그룹 단위로 대상을 선택할 수 있다(현재 범위는 Equipment 그룹만 지원).
  • FR-5.3 Recipe 그룹과 Equipment 그룹은 서로 다른 개념이며, UI에서 명확히 구분 표시되어야 한다.
04

비기능 요구사항

■ NFR-1. 권한

  • NFR-1.1 그룹 생성, 멤버 편집, 이름 변경, 삭제는 Vendor, Admin만 수행 가능하다.
  • NFR-1.2 일반 사용자는 그룹 정보를 조회할 수 있으나 변경할 수 없다.

■ NFR-2. 데이터 무결성

  • NFR-2.1 그룹 변경 이력(생성·수정·삭제)은 자동 기록되며 수정·삭제할 수 없다.
  • NFR-2.2 그룹 삭제 시 소속 Recipe는 절대 함께 삭제되지 않는다.

■ NFR-3. UI 일관성

  • NFR-3.1 Recipe 그룹과 Equipment 그룹의 관리 화면은 동일한 UX 패턴을 따른다(생성·편집·삭제 동작 흐름).
  • NFR-3.2 Recipe 그룹 노드와 Equipment 그룹 노드는 시각적으로 구분 가능하도록 표시된다.
05

제약사항 · 정책

  • 정책 Recipe는 여러 그룹에 동시 소속될 수 있다. 공정·용도·장비 등 다축 분류가 필요하기 때문이다. (Equipment 그룹의 단일 소속 정책과 의도적으로 다르다.)
  • 정책 그룹 삭제 시 소속 Recipe는 보존된다. Recipe는 다른 그룹 소속을 유지하거나 "미지정" 상태로 복귀한다.
  • 정책 그룹명 중복 검사는 대소문자를 구분하지 않는다. (Equipment 그룹과 동일 정책)
  • 정책 Recipe 그룹 변경 이력은 자동 기록되며 수정·삭제할 수 없다.
  • 제약 그룹 관리 권한은 Vendor, Admin로 한정된다.
  • 제약 Recipe 그룹과 Equipment 그룹은 별개의 데이터 모델이며 상호 변환되지 않는다.
06

관련 Test Cases

  • TC_RCP_GRP_001 — Recipe 그룹 생성 (FR-1)
  • TC_RCP_GRP_002 — 그룹명 중복 차단 (FR-1.3, FR-1.4)
  • TC_RCP_GRP_003 — 그룹에 Recipe 추가 · 제거 (FR-2)
  • TC_RCP_GRP_004 — 그룹 이름 변경 (FR-3)
  • TC_RCP_GRP_005 — 그룹 삭제 (소속 Recipe 처리 방식 확인) (FR-4, NFR-2.2)
  • TC_RCP_GRP_006 — Recipe가 다중 그룹에 속할 수 있는지 동작 확인 (FR-2.4)
