URS_RCP_MGR

Recipe 조회 · 관리

User Requirement Core Function
01

기능 개요

서버에 등록된 모든 Recipe를 한 화면에서 조회·검색·관리하는 RMS의 진입 화면이다. 사용자는 이 화면을 통해 어떤 Recipe가 어떤 버전으로 존재하는지 파악하고, 상세 파라미터를 확인하며, 다음 행위(비교·배포·삭제)를 시작한다.

Recipe Manager는 RMS 사용자가 가장 자주 머무는 화면이며, 다른 모든 Recipe 관련 화면(Compare, Deploy, Versions, Direct)으로의 출발점 역할을 한다.

02

사용자 니즈

주요 사용자

  • 공정 엔지니어 — 특정 Recipe의 현재 파라미터를 빠르게 확인
  • 장비 엔지니어 — 어떤 장비에 어떤 Recipe가 있는지 추적
  • 공정 표준화 팀 — 전체 Recipe 자산을 한눈에 조감

Pain Point (RMS 도입 전)

  • 장비별로 흩어진 Recipe를 일일이 들여다봐야 함
  • 같은 이름의 Recipe라도 장비마다 내용이 다른 경우가 흔함 (버전 혼선)
  • "이 Recipe가 마지막으로 변경된 게 언제더라?"가 안 보임
  • Recipe 수가 늘어날수록 어디 있는지 찾기조차 어려움

기대 효과

  • 모든 Recipe가 단일 화면에서 트리·목록 형태로 즉시 조회
  • 버전별 변경 시각·변경자를 통한 이력 가시화
  • 검색·정렬·필터로 대량 Recipe 환경에서도 신속한 탐색
03

기능 요구사항

■ FR-1. Recipe 목록 조회

  • FR-1.1 화면 진입 시 서버에 등록된 모든 Recipe가 좌측 트리 형태로 표시되어야 한다.
  • FR-1.2 각 Recipe 노드 하위에는 보유한 모든 버전이 펼침/접힘 가능한 형태로 표시되어야 한다.
  • FR-1.3 각 버전에는 버전 번호와 수집/등록 일시가 함께 표시되어야 한다.
  • FR-1.4 Master로 지정된 버전은 별도 아이콘 또는 배지로 시각적으로 구분되어야 한다.

■ FR-2. 상세 정보 표시

  • FR-2.1 Recipe 또는 버전 선택 시 중앙 영역에 Recipe Information(타입, 버전, 최종 수정일, 수집 장비, 설명)이 표시되어야 한다.
  • FR-2.2 우측 영역에 Recipe Contents(파라미터 목록)가 표시되어야 한다.
  • FR-2.3 다른 버전으로 전환 시 두 영역이 즉시 갱신되어야 한다.
  • FR-2.4 표시되는 파라미터는 Manage Recipe Parameters의 표시/숨김 설정에 따른다.
  • FR-2.5 Recipe 출처(Collected / Custom XML / Direct Upload)를 시각적으로 구분 표시한다.

■ FR-3. 검색 · 정렬 · 필터

  • FR-3.1 사용자는 Recipe 이름으로 실시간 검색할 수 있어야 한다. 대소문자를 구분하지 않으며, 부분 일치를 지원한다.
  • FR-3.2 검색어 입력 시 매칭된 문자열은 시각적으로 강조 표시된다.
  • FR-3.3 사용자는 Recipe 목록을 이름 또는 수집일 기준으로 정렬할 수 있어야 한다.
  • FR-3.4 같은 Recipe 내 버전은 기본적으로 최신순(내림차순)으로 정렬된다.
  • FR-3.5 정렬 기본값은 수집일 내림차순이며, 가장 최근 변경된 Recipe가 상단에 위치한다.
  • FR-3.6 사용자는 특정 장비를 선택하여 해당 장비에서 수집된 Recipe만 필터링할 수 있어야 한다.

■ FR-4. Recipe 삭제

  • FR-4.1 사용자는 Recipe를 선택하여 삭제할 수 있어야 한다.
  • FR-4.2 삭제 전 확인 팝업에는 Recipe 이름, 보유 버전 수, 영향 안내가 명시되어야 한다.
  • FR-4.3 현재 어떤 장비에 배포 중인 Recipe를 삭제할 경우, 배포된 장비 목록을 명시한 경고를 표시한 후 사용자 확인을 받는다.
  • FR-4.4 삭제는 Soft Delete로 처리되며, 30일간 복원 가능해야 한다.
  • FR-4.5 삭제 이력은 Recipe History에 자동 기록된다.

■ FR-5. 다른 화면으로의 액션 진입

  • FR-5.1 우측 액션 영역에서 Compare, Deploy, Download, Upload, Delete 버튼을 제공한다.
  • FR-5.2 액션 버튼은 사용자 권한과 Recipe 선택 상태에 따라 활성/비활성된다.
  • FR-5.3 Compare 진입 시 현재 선택된 Recipe가 비교 슬롯에 자동 입력되어야 한다.

■ FR-6. UI 상태 표시

  • FR-6.1 데이터 로딩 중에는 로딩 인디케이터를 300ms 이후 표시한다. (짧은 로딩에는 표시하지 않아 깜빡임 방지)
  • FR-6.2 등록된 Recipe가 없는 경우 Empty State 안내 메시지와 다음 액션 유도(Direct Upload 바로가기 등)를 제공한다.
  • FR-6.3 화면 진입 시 마지막으로 열었던 Recipe를 자동 선택한다.
04

비기능 요구사항

■ NFR-1. 성능

  • NFR-1.1 일반 환경(Recipe 100개 이하)에서 화면 진입 후 5초 이내에 모든 Recipe 목록이 표시되어야 한다.
  • NFR-1.2 대용량 환경(Recipe 500개 이상)에서도 5초 이내 초기 로딩이 가능해야 한다. 클라이언트 가상 스크롤 + 서버 페이지네이션 적용.
  • NFR-1.3 Recipe 선택 후 상세 정보 표시까지 1.5초 이내.
  • NFR-1.4 검색 입력 시 응답 지연은 300ms 이내. (디바운스 150~300ms 권장)
  • NFR-1.5 운영 SLA 기준 1 서버당 Recipe 최대 2,000개를 지원한다.

■ NFR-2. 권한

  • NFR-2.1 모든 사용자(All Roles)는 Recipe 조회·검색·정렬·필터링이 가능해야 한다.
  • NFR-2.2 Recipe 삭제는 Vendor, Admin만 수행 가능하다. Engineer 이하는 Delete 버튼이 비활성 또는 숨김 처리된다.
  • NFR-2.3 권한 검증은 UI 제약과 서버 API 이중 방어로 수행되어야 한다.

■ NFR-3. 사용성

  • NFR-3.1 좌측 트리 스크롤은 60fps 기준으로 끊김이 없어야 한다.
  • NFR-3.2 검색은 Recipe 이름 외 Description, Collected Equipment 필드도 부분 매칭으로 검색 가능해야 한다.
05

제약사항 · 정책

  • 정책 화면 진입 시 마지막으로 선택했던 Recipe가 자동으로 다시 선택된다. (사용자별 마지막 선택 상태 보존)
  • 정책 정렬 기본값은 수집일 내림차순이다. 최근 변경된 Recipe가 상단에 표시되어 운영 가시성을 확보한다.
  • 정책 버전 내 정렬은 최신 버전이 상단(내림차순)이다.
  • 정책 Recipe 삭제는 Soft Delete 방식으로 처리하며 30일간 복원 가능하다. 30일 경과 후 자동 영구 삭제된다.
  • 정책 현재 배포 중인 Recipe도 삭제 가능하나, 배포된 장비 목록을 명시한 경고 후 사용자 확인을 받는다.
  • 제약 Recipe 삭제 권한은 Vendor, Admin에 한정된다.
  • 제약 1 서버당 Recipe 최대 2,000개 지원. 초과 시 추가 서버 분리 또는 보존 정책 검토가 필요하다.
  • 제약 Recipe 출처는 Collected, Custom XML, Direct Upload 세 가지로만 분류한다. 각 출처는 시각 아이콘으로 구분 표시한다.
