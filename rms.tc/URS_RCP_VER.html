URS_RCP_VER

Recipe 버전 관리

User Requirement Core Function
01

기능 개요

한 Recipe가 보유한 여러 버전을 조회·관리하고, 그중 하나를 Master Recipe로 지정하는 기능이다. Master Recipe는 Deploy Recipes의 기본 선택값과 Compare Recipes의 기준 버전 등으로 사용되며, 해당 Recipe의 "운영 표준"을 의미한다.

Recipe Manager가 Recipe 단위의 관리라면, Manage Recipe Versions는 같은 Recipe 내부의 시간축을 다루는 화면이다. 어떤 버전이 표준인지, 어떤 변경을 거쳐 왔는지를 한곳에서 관리한다.

02

사용자 니즈

주요 사용자

  • 공정 표준화 팀 — 어떤 버전을 Master로 운영할지 결정
  • 공정 엔지니어 — Recipe의 변경 이력을 시간순으로 확인
  • 장비 엔지니어 — 잘못된 버전 배포 후 빠른 롤백

Pain Point (RMS 도입 전)

  • Recipe 변경 후 어떤 버전이 "현재 표준"인지 명확히 표시되지 않음
  • 버전 변경 사유가 개인 메모에만 남아 사후 추적이 어려움
  • 롤백이 필요한 경우 이전 버전을 다시 만들거나 백업에서 복원해야 함

기대 효과

  • Master 버전이 명확히 표시되어 운영 표준이 가시화됨
  • 버전별 변경 사유와 변경자가 함께 기록되어 이력 추적 가능
  • 이전 버전을 즉시 Master로 재지정하는 방식의 빠른 롤백 가능
03

기능 요구사항

■ FR-1. 버전 목록 조회

  • FR-1.1 사용자는 선택한 Recipe의 모든 버전을 목록 형태로 조회할 수 있어야 한다.
  • FR-1.2 각 버전 행에는 버전 번호, 수집/등록 일시, 변경자, 코멘트, Master 여부가 표시되어야 한다.
  • FR-1.3 Master 버전은 별도 아이콘 또는 배지로 시각적으로 구분된다.
  • FR-1.4 각 버전 행에 현재 배포된 장비 수가 함께 표시된다.
  • FR-1.5 기본 정렬은 최신 버전이 상단(내림차순)이다.

■ FR-2. Master Recipe 지정

  • FR-2.1 사용자는 특정 버전을 Master로 지정할 수 있어야 한다.
  • FR-2.2 새로운 버전을 Master로 지정하면 이전 Master는 자동으로 해제된다. 한 Recipe당 Master는 항상 1개이다.
  • FR-2.3 Master 변경은 트랜잭션으로 처리되어, 이전 Master 해제와 새 Master 지정이 원자적으로 수행된다.
  • FR-2.4 Master 지정 시 변경 사유를 입력하는 선택 필드를 제공한다.

■ FR-3. Master 변경 이력

  • FR-3.1 Master 변경은 Recipe History에 자동 기록된다.
  • FR-3.2 기록 항목에는 변경자, 변경 시각, 이전 Master, 새 Master, 변경 사유(입력된 경우)가 포함된다.
  • FR-3.3 Master 변경 이력은 수정·삭제할 수 없다(append-only).
  • FR-3.4 사용자는 Master 변경 이력을 별도 뷰에서 시계열로 조회할 수 있어야 한다.

■ FR-4. 버전 코멘트 관리

  • FR-4.1 각 버전에는 자유 텍스트 코멘트(Description)를 입력할 수 있어야 한다.
  • FR-4.2 코멘트 변경 이력이 보존되어 어떤 변경이 있었는지 추적 가능해야 한다.
  • FR-4.3 코멘트는 Recipe Manager의 상세 정보 영역에서도 표시된다.

■ FR-5. 버전 삭제

  • FR-5.1 사용자는 일반 버전(Master 아님)을 삭제할 수 있다.
  • FR-5.2 Master 버전은 삭제할 수 없다. 삭제하려면 먼저 다른 버전을 Master로 재지정해야 한다.
  • FR-5.3 마지막 남은 버전을 삭제하려는 경우, Recipe 자체 삭제 여부를 사용자에게 확인한다.
  • FR-5.4 버전 삭제는 Soft Delete로 처리되며, 30일간 복원 가능하다.
  • FR-5.5 버전 삭제 이력은 자동 기록된다.
04

비기능 요구사항

■ NFR-1. 데이터 무결성

  • NFR-1.1 Master 변경은 DB 트랜잭션으로 처리되어, 이전 Master 해제와 새 Master 지정이 모두 성공해야만 커밋된다.
  • NFR-1.2 두 사용자가 동시에 다른 버전을 Master로 지정하는 경우, 마지막 쓰기가 승리하며 다른 사용자에게 변경 알림이 표시된다.
  • NFR-1.3 한 Recipe에 Master 지정 버전은 항상 0개 또는 1개로 제약된다.

■ NFR-2. 권한

  • NFR-2.1 Master 지정은 Vendor, Admin, Engineer가 수행 가능하다.
  • NFR-2.2 버전 삭제는 Vendor, Admin만 수행 가능하다.
  • NFR-2.3 버전 코멘트 작성은 해당 버전을 등록한 사용자 외에 Admin 이상도 수정 가능하다.

■ NFR-3. 가시성

  • NFR-3.1 Master 표시는 Recipe Manager 트리, Manage Recipe Versions 목록, Dashboard 등 모든 관련 화면에서 일관되게 적용되어야 한다.
05

제약사항 · 정책

  • 정책 한 Recipe당 Master 버전은 항상 1개 이하이다. 새 Master 지정 시 이전 Master는 자동 해제된다.
  • 정책 Master 버전은 삭제할 수 없다. 삭제하려면 먼저 다른 버전을 Master로 재지정해야 한다.
  • 정책 마지막 남은 버전을 삭제하면 Recipe 자체가 삭제된다. 사용자에게 확인 후 진행한다.
  • 정책 버전 삭제는 Soft Delete로 처리하며 30일간 복원 가능하다.
  • 정책 Master 변경 이력은 수정·삭제할 수 없다(append-only).
  • 제약 Master 지정 권한은 Vendor, Admin, Engineer로 한정된다.
  • 제약 버전 삭제 권한은 Vendor, Admin로 한정된다.
