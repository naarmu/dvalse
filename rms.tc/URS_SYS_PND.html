URS_SYS_PND

Pending Recipes 알림

User Requirement Notification
01

기능 개요

장비에서 새로 수집되었지만 아직 RMS 서버에 정식 등록되지 않은 Recipe·Method가 있을 때, 사용자가 이를 인지하고 등록 여부를 결정할 수 있도록 안내하는 화면이다. 사용자 로그인 시 Pending 항목이 있으면 자동으로 팝업이 표시된다.

Pending Recipes는 자동 수집 결과를 그대로 RMS 자산에 편입시키지 않고, 사용자 검토 단계를 한 번 거치게 함으로써 의도하지 않은 Recipe가 자산에 섞이는 것을 방지한다.

02

사용자 니즈

주요 사용자

  • RMS 관리자 — 신규 수집된 Recipe·Method의 정식 등록 결정
  • 장비 엔지니어 — 자신이 작업한 Recipe가 서버에 잘 올라왔는지 확인

Pain Point (RMS 도입 전)

  • 장비에서 만든 임시·실험용 Recipe까지 자동으로 자산에 섞여 들어옴
  • 새로 수집된 Recipe가 있는지 별도 화면에서 일일이 확인해야 함
  • 의도하지 않은 Recipe가 정식 자산이 되어 운영 혼선 발생

기대 효과

  • 로그인 직후 신규 Recipe·Method 인지
  • 사용자 검토 후 등록되어 의도하지 않은 자산 편입 방지
  • 나중으로 미루기·일괄 등록 등 유연한 처리 가능
03

기능 요구사항

■ FR-1. 자동 표시 조건

  • FR-1.1 사용자가 RMS에 로그인하는 시점에 Pending 항목이 1개 이상 존재하면 팝업이 자동 표시된다.
  • FR-1.2 Operator 권한 사용자에게는 팝업이 표시되지 않으며, Pending 처리 권한도 부여하지 않는다.
  • FR-1.3 팝업이 닫힌 이후에도 메인 화면 상단 알림 영역에서 Pending 항목 수를 확인할 수 있어야 한다.

■ FR-2. Pending 목록 표시

  • FR-2.1 팝업에 모든 Pending 항목이 목록으로 표시되어야 한다.
  • FR-2.2 각 항목에는 장비명, Recipe·Method 이름, 변경 유형(신규 / 변경 / 삭제 추정), 수집 시각이 함께 표시된다.
  • FR-2.3 사용자는 항목을 선택하여 상세 정보(파라미터 등)를 확인할 수 있다.

■ FR-3. 항목 처리

  • FR-3.1 사용자는 개별 항목을 선택하여 서버에 정식 등록할 수 있어야 한다.
  • FR-3.2 사용자는 다수 항목을 동시 선택하여 일괄 등록할 수 있어야 한다.
  • FR-3.3 사용자는 특정 항목을 무시(등록하지 않고 Pending 목록에서 제외)할 수 있어야 한다.
  • FR-3.4 사용자는 팝업을 "나중에" 옵션으로 닫을 수 있다. 이 경우 항목은 Pending 목록에 그대로 남는다.
  • FR-3.5 "무시"와 "나중에"의 차이가 사용자에게 명확히 전달되어야 한다.

■ FR-4. 일괄 처리 시 부분 실패

  • FR-4.1 다수 항목 일괄 등록 중 일부 실패 시, 성공한 항목은 등록되고 실패한 항목만 Pending 목록에 남는다.
  • FR-4.2 실패 사유가 항목별로 표시되어야 한다.
  • FR-4.3 사용자는 실패한 항목만 선택하여 재시도할 수 있다.

■ FR-5. 이력 기록

  • FR-5.1 Pending 항목의 등록·무시·실패는 모두 RMS Log 또는 Recipe/Method History에 자동 기록된다.
  • FR-5.2 처리자, 처리 시각, 처리 결과가 함께 기록된다.
04

비기능 요구사항

■ NFR-1. 권한

  • NFR-1.1 Pending 처리 권한은 Vendor, Admin, Engineer로 한정된다.
  • NFR-1.2 Operator는 팝업 자체를 보지 않으며 Pending 항목에 영향을 주지 않는다.
  • NFR-1.3 권한 검증은 UI 제약과 서버 API 이중 방어로 수행된다.

■ NFR-2. 사용성

  • NFR-2.1 항목 수가 많은 경우(50건 이상) 팝업이 화면을 가리지 않도록 스크롤 가능한 형태로 표시된다.
  • NFR-2.2 "무시"와 "나중에" 라벨은 의미가 명확히 구분되도록 표기한다.

■ NFR-3. 데이터 무결성

  • NFR-3.1 두 사용자가 동시에 같은 Pending 항목을 처리하는 경우, 마지막 쓰기가 승리하며 다른 사용자에게 처리 결과가 표시된다.
  • NFR-3.2 "무시" 처리한 항목은 동일 수집 시 다시 Pending에 나타나지 않는다(중복 알림 방지).
05

제약사항 · 정책

  • 정책 Pending 항목은 자동으로 자산에 편입되지 않는다. 사용자 검토 단계를 항상 거친다.
  • 정책 팝업 자동 표시 시점은 로그인 직후이다. 사용자가 작업 도중 새 항목 알림으로 방해받지 않도록 한다.
  • 정책 "무시"는 영구 처리(다시 안 나타남), "나중에"는 일시 처리(다음 로그인 시 다시 표시)로 의미를 분리한다.
  • 정책 Operator 권한 사용자에게는 Pending 팝업을 표시하지 않는다. Pending 처리는 운영 결정이 필요한 행위이기 때문이다.
  • 정책 일괄 처리 시 일부 실패가 발생해도 성공한 항목은 정상 등록된다(부분 성공 허용).
  • 제약 Pending 처리 권한은 Vendor, Admin, Engineer로 한정된다.
  • 제약 모든 Pending 처리(등록·무시·실패) 이력은 자동 기록되며 수정·삭제할 수 없다.
06

관련 Test Cases

이 URS의 요구사항을 검증하는 QA 테스트 케이스 6건:

  • TC_SYS_PND_001 — Pending Recipes 팝업 자동 표시 조건 확인 (FR-1)
  • TC_SYS_PND_002 — Pending 목록의 장비 · Recipe명 · 변경유형 표시 (FR-2)
  • TC_SYS_PND_003 — Pending 항목 선택 후 서버 등록 실행 (FR-3.1)
  • TC_SYS_PND_004 — Pending 항목 무시 · 나중에 닫기 동작 (FR-3.3, FR-3.4, FR-3.5)
  • TC_SYS_PND_005 — 다수 Pending 항목 일괄 등록 (FR-3.2)
  • TC_SYS_PND_006 — Pending 등록 중 실패 항목 처리 (FR-4)
