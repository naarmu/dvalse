URS_HST_DEP

배포 이력

User Requirement Audit
01

기능 개요

RMS에서 수행된 모든 배포 작업(Recipe·Method·Custom XML)의 이력을 조회하는 화면이다. Deploy Recipes·Deploy Methods·Deploy Custom XML 화면에서 발생한 배포 동작이 모두 이 화면에 통합 기록된다.

Deployment Log는 RMS 운영 활동 중 가장 중요한 감사 대상이다. 어떤 자료가 언제 어느 장비에 배포되었고 그 결과가 어땠는지를 모두 추적 가능해야 한다.

02

사용자 니즈

주요 사용자

  • 공정 표준화 팀 — 배포 활동 추적
  • 장비 엔지니어 — 배포 실패 원인 분석
  • 감사 담당자 — 공정 변경 이력 검증

Pain Point (RMS 도입 전)

  • 배포가 언제 누구에 의해 수행되었는지 기록되지 않음
  • 실패한 배포의 원인 추적이 어려움
  • 감사 대응 시 배포 자료 수집에 많은 시간 소요

기대 효과

  • 모든 배포 활동의 자동 기록
  • 실패 건의 원인 즉시 확인
  • 감사 자료의 즉시 추출
03

기능 요구사항

■ FR-1. 배포 이력 전체 조회

  • FR-1.1 화면 진입 시 모든 배포 이력이 시계열(최신 우선)로 표시된다.
  • FR-1.2 각 항목에는 배포 시각, 배포자, Type(Recipe / Method / Custom XML), 자료명·버전, 대상 장비, 결과(성공·실패), 실패 시 원인이 표시된다.
  • FR-1.3 다수 장비에 동시 배포한 경우, 장비별로 각각 레코드가 생성되어 표시된다.
  • FR-1.4 페이지네이션 또는 가상 스크롤로 대량 데이터를 효율적으로 표시한다.

■ FR-2. Type별 필터

  • FR-2.1 사용자는 Type(Recipe / Method / Custom XML)으로 필터링할 수 있어야 한다.
  • FR-2.2 Type 필터는 시각적으로 구분되는 색상·아이콘으로 표시된다.

■ FR-3. 결과별 필터

  • FR-3.1 사용자는 결과(성공·실패)로 필터링할 수 있어야 한다.
  • FR-3.2 실패 건만 필터링하여 운영 문제 추적이 용이해야 한다.

■ FR-4. 실패 건 상세 원인 조회

  • FR-4.1 사용자는 실패 항목을 선택하여 상세 원인을 확인할 수 있어야 한다.
  • FR-4.2 실패 원인은 기술적 상세 + 사용자 친화 설명 + 해결 힌트의 세 층위로 제공된다.
  • FR-4.3 실패 항목에서 "원클릭 재시도" 액션을 제공한다.
  • FR-4.4 재시도 이력은 원 실패 이력과 별도 레코드로 기록된다.

■ FR-5. Export

  • FR-5.1 사용자는 현재 필터링 결과를 CSV 또는 Excel로 Export할 수 있어야 한다.
  • FR-5.2 Export 파일은 SEMI 감사 대응 포맷에 맞춰 컬럼 구성과 메타가 포함된다.

■ FR-6. 데이터 보존

  • FR-6.1 배포 이력은 어떤 사용자도 수정·삭제할 수 없다(append-only).
  • FR-6.2 이력 보존 기간은 최소 3년이다.
  • FR-6.3 삭제된 Recipe·Method·장비의 배포 이력도 보존된다.
04

비기능 요구사항

■ NFR-1. 성능

  • NFR-1.1 화면 진입 후 초기 데이터 표시까지 5초 이내.
  • NFR-1.2 필터 적용 응답은 3초 이내.

■ NFR-2. 권한

  • NFR-2.1 Deployment Log 조회는 모든 사용자(All Roles)가 가능하다.
  • NFR-2.2 Export는 Vendor, Admin, Engineer가 가능하다.
  • NFR-2.3 재시도 액션은 원 배포 권한과 동일하게 Vendor, Admin, Engineer로 한정된다.

■ NFR-3. 데이터 무결성

  • NFR-3.1 이력은 append-only로 관리된다.
  • NFR-3.2 삭제된 자료의 배포 이력도 보존된다.
05

제약사항 · 정책

  • 정책 Deployment Log는 Recipe·Method·Custom XML의 모든 배포를 통합 기록한다. Type 컬럼으로 구분 표시한다.
  • 정책 다수 장비 동시 배포 시 장비별로 별도 레코드를 생성한다. 장비별 독립 추적이 가능해야 한다.
  • 정책 배포 이력은 append-only로 관리된다. 어떤 사용자도 수정·삭제할 수 없다.
  • 정책 이력 보존 기간은 최소 3년이다.
  • 정책 실패 항목에 원클릭 재시도 액션을 제공한다. 재시도 이력은 원 실패 이력과 별도 레코드로 기록한다.
  • 정책 실패 원인은 기술적 상세 + 사용자 친화 설명 + 해결 힌트의 3층위로 제공한다.
  • 정책 삭제된 자료(Recipe·Method·장비)의 배포 이력도 보존된다.
  • 제약 Export는 Vendor, Admin, Engineer로 한정된다.
06

  • TC_HST_DEP_003 — 성공 · 실패 상태별 필터 (FR-3)
  • TC_HST_DEP_004 — 배포 실패 건 상세 원인 조회 (FR-4)
