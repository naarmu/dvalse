URS_SYS_LGN

로그인 · 인증

User Requirement Security
01

기능 개요

RMS Client 실행 시 사용자가 자신의 계정으로 인증하는 화면이다. 로그인이 성공해야만 RMS의 기능을 사용할 수 있으며, 사용자의 Role에 따라 사용 가능한 기능 범위가 결정된다.

로그인은 RMS 사용의 시작점이자 권한 검증의 출발점이다. 모든 사용자 행위가 로그인된 계정 정보를 기준으로 이력에 기록된다.

02

사용자 니즈

주요 사용자

  • RMS의 모든 사용자 (Vendor, Admin, Engineer, Operator)

Pain Point (RMS 도입 전)

  • 장비별 로컬 인증으로 인해 동일인이 여러 계정을 갖고 있음
  • 장비 간 권한 정책이 일관되지 않음
  • 누가 어떤 작업을 했는지 통합 추적이 어려움

기대 효과

  • RMS 단일 계정으로 모든 RMS 기능 사용
  • Role 기반의 일관된 권한 제어
  • 모든 작업이 사용자별로 이력에 기록되어 추적 가능
03

기능 요구사항

■ FR-1. 로그인 입력

  • FR-1.1 사용자는 ID와 비밀번호를 입력하여 로그인할 수 있어야 한다.
  • FR-1.2 비밀번호 입력 필드는 기본적으로 마스킹 처리되며, 사용자가 토글로 표시 여부를 변경할 수 있어야 한다.
  • FR-1.3 ID 또는 비밀번호가 비어 있는 경우 로그인 버튼이 비활성 또는 검증 에러로 차단된다.
  • FR-1.4 직전에 로그인한 ID는 입력 필드에 자동 채워질 수 있다(설정 가능).

■ FR-2. 인증 처리

  • FR-2.1 입력된 ID·비밀번호가 서버에 등록된 정보와 일치하면 로그인이 성공한다.
  • FR-2.2 일치하지 않는 경우 로그인이 실패하며, "ID 또는 비밀번호가 올바르지 않습니다." 형태의 메시지가 표시된다. (어느 쪽이 틀렸는지는 알리지 않는다.)
  • FR-2.3 Disabled 계정으로 로그인 시도 시 별도의 안내 메시지가 표시되며 로그인은 차단된다.
  • FR-2.4 비밀번호는 서버 전송 시 평문으로 전송되지 않으며, TLS 암호화 채널 위에서만 전송된다.

■ FR-3. 서버 미연결 처리

  • FR-3.1 서버에 연결되지 않은 상태에서 로그인 시도 시 명확한 에러 메시지가 표시되어야 한다.
  • FR-3.2 사용자는 즉시 Server Setup 화면으로 이동하여 서버 정보를 수정할 수 있어야 한다.

■ FR-4. 로그인 후 진입

  • FR-4.1 로그인 성공 시 사용자의 Role에 적합한 기본 화면으로 진입한다.
  • FR-4.2 진입 화면 정책: Vendor·Admin은 RMS Dashboard, Engineer는 Equipment Dashboard, Operator는 Recipe Manager.
  • FR-4.3 사용자가 마지막으로 머물던 화면을 기억하여 재로그인 시 그 화면으로 진입하는 옵션을 제공할 수 있다(설정 가능).

■ FR-5. 로그아웃

  • FR-5.1 사용자는 어느 화면에서든 로그아웃할 수 있어야 한다.
  • FR-5.2 로그아웃 시 사용자 세션이 즉시 종료되며, 진행 중이던 작업이 있으면 사용자에게 확인을 받는다.
  • FR-5.3 로그아웃 후 Login 화면으로 돌아간다.
04

비기능 요구사항

■ NFR-1. 보안

  • NFR-1.1 비밀번호는 서버에 평문으로 저장되지 않는다. 단방향 해시(예: bcrypt) + Salt 적용을 원칙으로 한다.
  • NFR-1.2 일정 횟수 이상 로그인 실패 시 해당 계정에 대한 추가 시도를 일정 시간 차단한다.
  • NFR-1.3 로그인 시도(성공·실패) 모두 RMS Log에 기록된다.
  • NFR-1.4 로그인 메시지에서 ID·비밀번호 중 어느 쪽이 틀렸는지 구분하지 않는다(정보 노출 방지).

■ NFR-2. 성능

  • NFR-2.1 정상 환경에서 로그인 응답은 2초 이내.
  • NFR-2.2 로그인 후 초기 화면 진입까지 5초 이내.

■ NFR-3. 사용성

  • NFR-3.1 로그인 실패 메시지는 사용자가 다음 행동을 결정할 수 있도록 명확해야 한다.
  • NFR-3.2 키보드만으로 모든 로그인 동작이 가능해야 한다(접근성).
05

제약사항 · 정책

  • 정책 로그인 실패 메시지는 ID·비밀번호 중 어느 쪽이 틀렸는지 구분하지 않는다. 계정 존재 여부 노출을 방지하기 위함이다.
  • 정책 Disabled 계정에 대해서는 별도 메시지를 표시한다. 일반 인증 실패와 구분되어야 사용자가 관리자에게 적절히 문의할 수 있다.
  • 정책 로그인 후 진입 화면은 Role별로 다르게 설정된다. 사용자에게 가장 자주 쓰이는 화면을 기본값으로 한다.
  • 정책 로그인 시도 이력(성공·실패 모두)은 RMS Log에 기록되며 수정·삭제할 수 없다.
  • 제약 비밀번호는 평문으로 저장·전송되지 않는다. 저장은 단방향 해시, 전송은 TLS.
  • 제약 일정 횟수 이상 연속 로그인 실패 시 일시 차단한다. 무차별 대입 공격 방어가 목적이다.
06

관련 Test Cases

이 URS의 요구사항을 검증하는 QA 테스트 케이스 7건:

  • TC_SYS_LGN_001 — 유효한 계정으로 로그인 성공 (FR-1, FR-2.1)
  • TC_SYS_LGN_002 — 잘못된 ID · PW 입력 시 에러 메시지 표시 (FR-2.2, NFR-1.4)
  • TC_SYS_LGN_003 — 빈 ID 또는 빈 PW 제출 시 검증 (FR-1.3)
  • TC_SYS_LGN_004 — 서버 미연결 상태에서 로그인 시도 (FR-3)
  • TC_SYS_LGN_005 — 로그인 후 Role별 초기 진입 화면 확인 (FR-4)
  • TC_SYS_LGN_006 — 로그아웃 동작 및 세션 종료 (FR-5)
  • TC_SYS_LGN_007 — 비활성 (Disabled) 계정 로그인 차단 (FR-2.3)
