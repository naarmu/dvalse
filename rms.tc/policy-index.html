01

권한·Role 정책

26건

중복 정책 묶음

9건 중복

Vendor·Admin 한정

  • 제약 URS_EQP_GRP 그룹 관리 권한은 Vendor, Admin로 한정된다.
  • 제약 URS_EQP_REG 등록·수정·삭제 권한은 Vendor, Admin로 한정된다.
  • 제약 URS_HST_RMS RMS Log 조회·Export 권한은 Vendor, Admin로 한정된다.
  • 제약 URS_HST_SVR Server Log 접근 권한은 Vendor, Admin로 한정된다.
  • 제약 URS_MTD_GRP 그룹 관리 권한은 Vendor, Admin로 한정된다.
  • 제약 URS_MTD_MGR Method 삭제 권한은 Vendor, Admin로 한정된다.
  • 제약 URS_RCP_GRP 그룹 관리 권한은 Vendor, Admin로 한정된다.
  • 제약 URS_RCP_VER 버전 삭제 권한은 Vendor, Admin로 한정된다.
  • 제약 URS_SET_CNN 연결 정보 변경 권한은 Vendor, Admin로 한정된다.
통합 제안

권한 매트릭스로 통합 권장: 9개 URS에서 "이 권한은 Vendor·Admin로 한정"이 반복된다. 별도의 권한 매트릭스 표(URS × Action × Role)로 한 번 정의하고 개별 URS에서는 참조만 한다.

7건 중복

Vendor·Admin·Engineer 한정

  • 제약 URS_HST_DEP Export는 Vendor, Admin, Engineer로 한정된다.
  • 제약 URS_HST_EQP 이력 조회는 모든 사용자에게 허용된다. Export는 Vendor, Admin, Engineer로 한정된다.
  • 제약 URS_HST_MTD Export는 Vendor, Admin, Engineer로 한정된다.
  • 제약 URS_HST_RCP Export는 Vendor, Admin, Engineer로 한정된다.
  • 제약 URS_MTD_DEP 배포 권한은 Vendor, Admin, Engineer로 한정된다.
  • 제약 URS_MTD_MGR Direct Upload 권한은 Vendor, Admin, Engineer로 한정된다. Direct Download는 모든 사용자에게 허용된다.
  • 제약 URS_RCP_DIR Direct Upload 권한은 Vendor, Admin, Engineer로 한정된다. Direct Download는 모든 사용자에게 허용된다.
통합 제안

권한 매트릭스로 통합 권장: 7개 URS에서 동일 패턴 반복. 위와 동일하게 권한 매트릭스로 통합.

2건 중복

Vendor 전용

  • 제약 URS_RCP_DEP 배포는 Engineer, Admin, Vendor만 수행 가능. Operator는 조회만 허용.
  • 정책 URS_SET_PRV Privilege 화면은 Vendor만 접근 가능하다. 보안 정책의 최상위 결정 권한이 명확히 분리되도록 한다.
통합 제안

권한 매트릭스로 통합 권장: 위와 동일.

개별 정책

  • 정책
    Vendor는 자기 자신의 권한을 축소할 수 없다. 시스템 잠금 방지가 목적이다.
    URS_SET_PRV · 권한 관리
  • 정책
    표시 설정 변경 권한은 Vendor, Admin으로 한정된다.
    URS_MTD_PRM · Method 파라미터 표시 설정
  • 정책
    현재 배포 중인 Recipe를 삭제할 경우, 경고 팝업 + 배포된 장비 목록 명시 후 삭제를 허용한다.
    URS_RCP_DEP · Recipe 배포
  • 제약
    Recipe 삭제 권한은 Vendor, Admin에 한정된다.
    URS_RCP_MGR · Recipe 조회 · 관리
  • 정책
    표시 설정 변경 권한은 Vendor, Admin으로 한정된다.
    URS_RCP_PRM · 파라미터 표시 설정
  • 제약
    업로드는 Vendor, Admin, Engineer만 수행 가능. Operator는 차단된다.
    URS_RCP_XML · Custom XML 배포
  • 정책
    Admin은 Engineer, Operator 계정만 관리할 수 있다. Vendor 계정 관리는 Vendor만 가능하다.
    URS_SET_ACC · 계정 관리
  • 제약
    Preference 설정은 데이터 무결성에 영향을 주지 않는 표시 옵션에 한정된다. 시스템 동작 관련 설정(연결 정보 등)은 별도 화면에서 관리한다.
    URS_SET_PRF · 개인 설정
02

이력·감사 (Audit)

23건

중복 정책 묶음

17건 중복

append-only

  • 정책 URS_CLT_RCP 수집 설정 변경(주기·트리거)은 자동 기록되며 수정·삭제할 수 없다.
  • 정책 URS_EQP_GRP 그룹 변경 이력은 자동 기록되며 수정·삭제할 수 없다.
  • 정책 URS_HST_DEP 배포 이력은 append-only로 관리된다. 어떤 사용자도 수정·삭제할 수 없다.
  • 정책 URS_HST_EQP 모든 이력은 append-only로 관리된다. 어떤 사용자도 수정·삭제할 수 없다.
  • 정책 URS_HST_MTD 모든 이력은 append-only로 관리된다.
  • 정책 URS_HST_RCP 모든 이력은 append-only로 관리된다.
  • 정책 URS_HST_RMS 이력은 append-only이며 어떤 경로에서도 수정·삭제할 수 없다.
  • 정책 URS_HST_SVR 서버 로그는 append-only로 관리된다.
  • 제약 URS_MTD_DEP Deployment Log는 어떤 사용자도 수정·삭제할 수 없다(append-only).
  • 정책 URS_MTD_GRP Method 그룹 변경 이력은 자동 기록되며 수정·삭제할 수 없다.
  • 제약 URS_RCP_DEP Deployment Log는 어떤 사용자도 수정·삭제할 수 없다. (append-only)
  • 정책 URS_RCP_GRP Recipe 그룹 변경 이력은 자동 기록되며 수정·삭제할 수 없다.
  • 정책 URS_RCP_VER Master 변경 이력은 수정·삭제할 수 없다(append-only).
  • 제약 URS_SET_ACC 모든 계정 변경 이력은 자동 기록되며 수정·삭제할 수 없다.
  • 정책 URS_SET_PRV 권한 변경 이력은 자동 기록되며 수정·삭제할 수 없다.
  • 정책 URS_SYS_LGN 로그인 시도 이력(성공·실패 모두)은 RMS Log에 기록되며 수정·삭제할 수 없다.
  • 제약 URS_SYS_PND 모든 Pending 처리(등록·무시·실패) 이력은 자동 기록되며 수정·삭제할 수 없다.
통합 제안

전사 공통 원칙으로 격상 권장: 17개 URS에서 동일 정책이 반복된다. 각 URS에는 "본 화면의 이력 데이터는 RMS의 공통 이력 정책을 따른다"로 간소화하고, 본 정책은 Documentation의 별도 항목으로 한 번만 정의.

6건 중복

3년 보존

  • 정책 URS_HST_DEP 이력 보존 기간은 최소 3년이다.
  • 정책 URS_HST_EQP 이력 보존 기간은 최소 3년이다. (반도체 업계 감사 요구사항 기준)
  • 정책 URS_HST_MTD 이력 보존 기간은 최소 3년이다.
  • 정책 URS_HST_RCP 이력 보존 기간은 최소 3년이다.
  • 정책 URS_HST_RMS 이력 보존 기간은 최소 3년이다.
  • 정책 URS_HST_SVR 보존 기간은 일반 로그 1년, Critical·Error 레벨 3년으로 차등 적용한다. 보존 비용과 감사 요구사항의 균형을 위함이다.
통합 제안

전사 공통 정책으로 격상 권장: 6개 URS에서 동일 보존 기간이 반복된다. "이력 데이터는 최소 3년 보존"을 공통 정책으로 정의하고, 예외(Server Log 일반 1년)만 개별 URS에 명시.

04

통신·보안

13건

중복 정책 묶음

4건 중복

TLS 필수

  • 제약 URS_CLT_MTD Agent–서버 통신은 TLS 암호화 채널 위에서만 허용된다.
  • 제약 URS_CLT_RCP Agent–서버 통신은 TLS 암호화 채널 위에서만 허용된다.
  • 제약 URS_SET_CNN Client–서버 통신은 TLS 암호화 채널 위에서만 허용된다.
  • 제약 URS_SYS_SVR Client–서버 통신은 TLS 암호화 채널 위에서만 허용한다.
통합 제안

전사 공통 보안 원칙으로 격상 권장: 4개 URS에서 반복. "RMS의 모든 통신(Client–Server, Agent–Server)은 TLS 암호화 채널 위에서 수행"을 시스템 보안 원칙으로 한 번 정의.

3건 중복

SHA-256 해시

  • 제약 URS_CLT_MTD 모든 전송에 SHA-256 해시가 동반되어야 한다.
  • 제약 URS_CLT_RCP 모든 전송에 SHA-256 해시가 동반되어야 한다(무결성 검증).
  • 정책 URS_RCP_DIR 업로드·다운로드 이력은 파일 해시값(SHA-256)을 포함해 기록된다. 사후 무결성 검증에 사용된다.
통합 제안

전사 공통 정책으로 격상 권장: 3개 URS에서 반복. "파일 전송 시 SHA-256 해시 동반"을 무결성 원칙으로 통합.

2건 중복

안티바이러스

  • 제약 URS_MTD_MGR 업로드 파일은 안티바이러스 스캔을 통과해야 서버에 등록된다.
  • 제약 URS_RCP_DIR 업로드 파일은 안티바이러스 스캔을 통과해야 서버에 등록된다.
통합 제안

업로드 보안 정책으로 통합 권장: 2개 URS에서 반복. "외부에서 들어오는 모든 파일(XML, PSR, Method)은 안티바이러스 스캔 통과 후 등록"으로 통합.

2건 중복

비밀번호 평문 금지

  • 제약 URS_SET_ACC 비밀번호는 단방향 해시 + Salt로 저장되며 어떤 화면에서도 평문 노출되지 않는다.
  • 제약 URS_SYS_LGN 비밀번호는 평문으로 저장·전송되지 않는다. 저장은 단방향 해시, 전송은 TLS.
통합 제안

전사 인증 정책으로 격상 권장: Login·Account 양쪽에서 반복. 인증·계정 관리 공통 정책으로 통합.

개별 정책

  • 제약
    XML 파서는 XXE 외부 엔티티 참조를 비활성화한 보안 설정으로만 동작한다.
    URS_RCP_XML · Custom XML 배포
  • 정책
    로그 상세 정보의 민감 데이터(비밀번호·토큰 등)는 자동 마스킹된다.
    URS_HST_SVR · 서버 시스템 로그
05

그룹·소속 정책

12건

중복 정책 묶음

4건 중복

Recipe·Method 다중 그룹

  • 정책 URS_EQP_GRP 한 장비는 단일 그룹에만 소속된다. (Recipe·Method 그룹의 다중 소속 정책과 의도적으로 다르다.) 장비 그룹은 운영 위계 표현에 가깝기 때문에 단일 소속이 적합하다.
  • 제약 URS_EQP_MGR 한 장비는 단일 그룹에만 소속된다. (Recipe·Method 그룹의 다중 소속 정책과 의도적으로 다르다.)
  • 정책 URS_MTD_GRP Method는 여러 그룹에 동시 소속될 수 있다. (Recipe 그룹과 동일 정책)
  • 정책 URS_RCP_GRP Recipe는 여러 그룹에 동시 소속될 수 있다. 공정·용도·장비 등 다축 분류가 필요하기 때문이다. (Equipment 그룹의 단일 소속 정책과 의도적으로 다르다.)
통합 제안

그룹 정책 매트릭스로 통합 권장: Recipe·Method 그룹 = 다중 소속, Equipment 그룹 = 단일 소속. 차이가 의도적이므로 한 표로 정리하면 명확.

3건 중복

그룹 삭제 시 자료 보존

  • 정책 URS_EQP_GRP 그룹 삭제 시 소속 장비는 보존된다. 장비는 "미지정" 상태로 복귀하며, 다른 그룹에 재배치하기 전까지 그룹 트리에서는 별도로 표시된다.
  • 정책 URS_MTD_GRP 그룹 삭제 시 소속 Method는 보존된다. Method는 다른 그룹 소속을 유지하거나 "미지정" 상태로 복귀한다.
  • 정책 URS_RCP_GRP 그룹 삭제 시 소속 Recipe는 보존된다. Recipe는 다른 그룹 소속을 유지하거나 "미지정" 상태로 복귀한다.
통합 제안

그룹 공통 동작으로 통합 권장: 3개 URS에서 동일. "그룹 삭제 시 소속 자료는 보존되며 미지정 상태로 복귀"를 공통 정책으로 통합.

3건 중복

그룹명 대소문자 무시

  • 정책 URS_EQP_GRP 그룹명 중복 검사는 대소문자를 구분하지 않는다.
  • 정책 URS_MTD_GRP 그룹명 중복 검사는 대소문자를 구분하지 않는다.
  • 정책 URS_RCP_GRP 그룹명 중복 검사는 대소문자를 구분하지 않는다. (Equipment 그룹과 동일 정책)
