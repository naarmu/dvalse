TC_RCP_MGR_009

Recipe 삭제 기능 (권한 · 확인 팝업)

Functional High
01

Precondition

  1. 삭제 대상 Recipe "TEST_TO_DELETE"가 서버에 존재 (버전 2개 포함)
  2. 해당 Recipe가 어떤 장비에도 현재 배포되어 있지 않음
02

Test Steps

  1. Recipe "TEST_TO_DELETE" 선택
  2. 우측 액션 영역의 Delete 버튼 클릭
  3. 확인 팝업 내용 확인 (버전 수 포함 여부 등)
  4. "확인" 선택
  5. Recipe가 목록에서 제거되고 History에 삭제 이력 기록 확인
  6. 동일 시나리오로 Operator 계정에서 시도 → Delete 버튼 비활성/숨김 확인
03

Test Data

Recipe: TEST_TO_DELETE (v1, v2 포함) Login 1: admin_test (Admin) — 삭제 성공 Login 2: operator_test (Operator) — 삭제 차단
04

Expected Result

  1. 확인 팝업에 Recipe 이름, 버전 수, 영향 안내가 명시됨
  2. "확인" 시 Recipe와 모든 버전이 서버에서 삭제
  3. Recipe History에 삭제 이력이 기록 (삭제자, 시각, 버전 수)
  4. "취소" 시 변경 없음
  5. Operator 계정에서는 Delete 버튼 자체가 비활성 또는 숨김
  6. 현재 어떤 장비에 배포된 Recipe는 삭제 시도 시 경고 또는 차단 (정책 확인 필요)
05

비고 (QA · 개발 메모)

정책 확정 필요: 현재 배포 중인 Recipe의 삭제를 허용할지 차단할지. 차단이 안전하지만 운영 편의성은 저하됨.

Master 버전 삭제: Master로 지정된 Recipe를 통째 삭제할 때 추가 확인이 필요할 수 있음.

06

제안 사항

정책 확정 현재 배포 중인 Recipe 삭제 정책이 필요합니다. 권장: "경고 후 허용 + 배포된 장비 목록 명시"로, 사용자가 충분히 인지한 상태에서 진행 가능하게.

데이터 무결성 Recipe를 완전 삭제(Hard Delete)보다는 소프트 삭제(Soft Delete, is_deleted 플래그)를 권장드립니다. 삭제 후 30일 복원 기능을 추가할 수 있고, 감사 이력도 온전히 유지됩니다.

UX 개선 파괴적 액션이므로 확인 팝업에서 "삭제" 버튼은 빨간색, 기본 포커스는 "취소"에 두는 것을 제안드립니다.
