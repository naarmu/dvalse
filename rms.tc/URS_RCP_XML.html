URS_RCP_XML

Custom XML 배포

User Requirement Core Function
01

기능 개요

장비 측에서 직접 작성하지 않은 Custom XML 파일을 서버에 업로드하고, 이를 장비로 배포하면 장비에서 자동으로 .psr Recipe 파일로 변환되는 기능이다. 일반적인 Recipe 운영 흐름(장비에서 수집 → 서버 등록 → 다시 배포)과 달리 장비를 거치지 않고도 새 Recipe를 생성할 수 있는 별도 경로를 제공한다.

RMS의 다른 기능들이 "기존 자산을 관리"하는 데 집중한다면, Deploy Custom XML은 "새로운 자산을 주입"하는 유일한 진입점이다. 이로 인해 보안·검증·이력 추적의 중요성이 상대적으로 더 높다.

흔히 Waferless Recipe Creation으로 불리는 운영 시나리오를 가능하게 하는 기능이다.

02

사용자 니즈

주요 사용자

  • 공정 표준화 팀 — 사내 표준 Recipe를 XML로 작성하여 여러 FAB의 장비에 일괄 배포
  • R&D 엔지니어 — 새로운 측정 시나리오를 장비에서 실험 없이 미리 정의
  • 장비 엔지니어 — 장비가 가동되기 전에 미리 Recipe를 준비

Pain Point (이 기능 도입 전)

  • 새 Recipe를 만들려면 반드시 장비 앞에서 직접 만들고 저장해야 함
  • 장비 가동 전이거나 정비 중일 때는 Recipe 작성이 불가능
  • 여러 장비에 동일 Recipe를 만들려면 각 장비를 일일이 방문해야 함
  • Recipe 생성 작업이 장비 가동 시간을 잠식

기대 효과

  • 장비 가용성과 무관하게 사무실에서 Recipe 정의 및 배포 가능
  • 표준 Recipe의 중앙 작성 → 다수 장비 일괄 배포로 표준화 시간 단축
  • 장비 가동 시간 절약 (Recipe 작성에 소요되던 비가동 시간 제거)
03

기능 요구사항

■ FR-1. XML 업로드

  • FR-1.1 사용자는 로컬 PC의 XML 파일을 파일 선택 또는 드래그 앤 드롭으로 업로드할 수 있어야 한다.
  • FR-1.2 업로드 진행률이 표시되어야 한다.
  • FR-1.3 업로드 완료 후 서버에 등록된 XML 목록에 추가되며, 메타 정보(파일명, 크기, 업로더, 일시)가 함께 표시된다.
  • FR-1.4 동일 파일명의 XML이 이미 존재하는 경우, 사용자에게 새 버전 추가 / 덮어쓰기 / 취소 옵션을 제공한다. 기본 선택값은 새 버전 추가이다.

■ FR-2. XML 유효성 검증

  • FR-2.1 업로드된 파일의 확장자가 .xml이 아닌 경우 클라이언트 단계에서 즉시 차단한다.
  • FR-2.2 서버는 업로드된 XML의 문법 유효성을 검증한다. 파싱 실패 시 명확한 에러 메시지(라인 번호 포함)를 제공한다.
  • FR-2.3 서버는 Recipe 스키마 적합성을 검증한다. 스키마 불일치 시 어떤 요소가 누락/오류인지 구체적으로 알려야 한다.
  • FR-2.4 검증 실패 시 파일은 서버에 저장되지 않으며, 사용자가 수정 후 재업로드할 수 있다.
  • FR-2.5 검증 실패 화면에서 사용자가 참고할 수 있는 예시 XML 다운로드 링크를 제공한다.

■ FR-3. 장비 배포

  • FR-3.1 사용자는 등록된 XML을 단일 장비 또는 다수 장비에 배포할 수 있어야 한다.
  • FR-3.2 사용자는 장비 그룹 단위로 일괄 배포할 수 있어야 한다.
  • FR-3.3 배포 진행 상태가 장비별로 개별 표시되어야 한다.
  • FR-3.4 배포 시작 전 사용자에게 추가 확인 단계를 제공한다. ("이 Recipe는 외부 출처입니다. 검증되었나요?")

■ FR-4. 장비 측 변환

  • FR-4.1 장비 Agent는 수신한 XML을 .psr 파일로 자동 변환한다.
  • FR-4.2 변환은 임시 파일에 먼저 기록 후 최종 rename하는 atomic 방식으로 수행되어야 한다. 변환 실패 시 장비의 기존 Recipe가 손상되지 않아야 한다.
  • FR-4.3 변환된 .psr은 장비의 정상 Recipe 경로에 저장되며, 일반 Recipe와 동일하게 장비에서 사용 가능하다.
  • FR-4.4 변환 성공 시 Agent가 변환된 .psr을 서버로 자동 수집한다.
  • FR-4.5 변환 실패 시 서버로 실패 보고가 전송되며, RMS UI에 알림이 표시된다. 실패 사유는 라인 번호 등 구체적 정보를 포함해야 한다.

■ FR-5. RMS 내 조회 · 후속 활용

  • FR-5.1 변환·수집된 Recipe는 Recipe Manager의 트리에 신규 Recipe로 표시된다.
  • FR-5.2 출처가 "Custom XML"임을 시각 아이콘으로 구분 표시한다.
  • FR-5.3 Custom XML 출신 Recipe도 일반 Recipe와 동일하게 Compare, Deploy, Direct Download 등 모든 기능을 사용할 수 있어야 한다.
  • FR-5.4 다수 장비에 동일 XML을 배포한 경우, 각 장비에서 변환된 .psr은 하나의 Recipe로 통합 관리하되 배포 이력은 장비별로 개별 기록된다.

■ FR-6. 이력 기록

  • FR-6.1 XML 업로드, 배포, 변환 성공/실패는 모두 별도 이력으로 기록된다.
  • FR-6.2 Deployment Log는 Recipe 배포와 Custom XML 배포를 Type 컬럼으로 구분 표시한다.
  • FR-6.3 모든 이력은 append-only로 관리된다.
04

비기능 요구사항

■ NFR-1. 보안

  • NFR-1.1 XML 파서는 외부 엔티티 참조(XXE)를 비활성화한다. 서버 측 파일 읽기 또는 SSRF 공격 방지가 필수이다.
  • NFR-1.2 클라이언트 확장자 검증, 서버 문법 검증, 서버 스키마 검증의 3단계 방어를 수행한다.
  • NFR-1.3 업로드 파일은 안티바이러스 스캔을 통과해야 서버에 등록된다.
  • NFR-1.4 업로드 권한은 Vendor, Admin, Engineer로 한정된다.

■ NFR-2. 데이터 무결성

  • NFR-2.1 XML → PSR 변환은 원자적(atomic)이어야 한다. 임시 파일 → rename 패턴 적용.
  • NFR-2.2 변환 실패 시 장비의 기존 Recipe는 절대 영향받지 않는다.
  • NFR-2.3 변환 실패한 XML은 원본 보존되어 추후 재시도 또는 분석이 가능해야 한다.

■ NFR-3. 성능 · 제약

  • NFR-3.1 XML 파일 최대 크기는 5MB로 제한한다.
  • NFR-3.2 다수 장비 동시 배포 시 변환 작업은 장비별로 병렬 수행된다.
  • NFR-3.3 변환 평균 소요 시간은 30초 이내를 목표로 한다.

■ NFR-4. 알림 · 가시성

  • NFR-4.1 변환 실패는 비동기 이벤트이므로, RMS UI에 실시간 알림(Toast 또는 알림 배지)으로 사용자에게 전달되어야 한다.
  • NFR-4.2 Custom XML 변환 실패 통계 위젯을 Dashboard에 제공하여 관리자가 실패 경향을 파악할 수 있다.
05

제약사항 · 정책

  • 정책 Custom XML로 생성된 Recipe는 Recipe Manager에서 Custom XML 출처 아이콘으로 시각 구분된다. 이는 검증 책임 주체가 사용자임을 명확히 하기 위함이다.
  • 정책 다수 장비에 동일 XML 배포 시, 변환된 Recipe는 RMS에서 단일 Recipe로 통합 관리한다. 배포 이력만 장비별로 분리 기록한다.
  • 정책 변환 실패 시 기존 동일명 Recipe는 보존되며, 관리자에게 알림이 발송된다.
  • 정책 동명 XML 업로드 시 기본 동작은 새 버전 추가이다. 덮어쓰기는 명시적 사용자 선택 시에만 수행된다.
  • 제약 XML 파일 최대 크기 5MB, 확장자 .xml만 허용.
  • 제약 업로드는 Vendor, Admin, Engineer만 수행 가능. Operator는 차단된다.
  • 제약 XML 파서는 XXE 외부 엔티티 참조를 비활성화한 보안 설정으로만 동작한다.
  • 제약 변환 실패한 XML은 자동 삭제되지 않고 별도 폴더에 보관되어 분석 가능하도록 한다.
