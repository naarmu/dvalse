TC_E2E_007

배포 중 장비 연결 끊김 발생 → 에러 처리 → 재배포 시 정상 완료 → 이력의 실패/성공 분리 기록

Negative Critical
01

Precondition

  1. Recipe X Master 버전 지정 완료
  2. Online 장비 4대 (EQ_001 ~ EQ_004)
  3. Engineer 이상 권한
02

Test Steps

  1. Deploy Recipes에서 Recipe X Master를 EQ_001 ~ EQ_004에 동시 배포 시작
  2. 배포 진행률이 30~70% 시점에 EQ_003의 네트워크를 강제 단절 (케이블 분리 또는 Agent 종료)
  3. 배포 진행 UI에서 EQ_003의 실패 표시 확인 (TC_RCP_DEP_005 연계)
  4. 배포 완료 후 결과 요약 확인: 성공 3, 실패 1 (TC_RCP_DEP_008 연계)
  5. Deployment Log에서 4건 각각 기록 확인 (성공 3 + 실패 1)
  6. 실패 건의 상세 원인 조회 (TC_HST_DEP_003 연계)
  7. EQ_003 네트워크 복구 후 온라인 상태 복귀 대기
  8. Deployment Log에서 실패 건 재시도 실행 (EQ_003만 대상)
  9. 재시도 성공 확인
  10. Deployment Log에 원 실패 1건 + 재시도 성공 1건이 별도 이력으로 기록되는지 확인 (append-only 검증)
  11. EQ_003의 Deployed Recipe가 정상 Recipe X로 표시되는지 확인
03

Test Data

초기 배포: 4대 장비 중간 단절: EQ_003 (30~70% 시점) 최종: 3 성공 + 1 실패 기록, 재시도로 EQ_003 복구
04

Expected Result

  1. 부분 실패가 전체 배포를 중단시키지 않음 (다른 장비는 성공 완료)
  2. 실패 장비의 기존 Recipe가 훼손되지 않음 (Atomic Write 확인)
  3. Deployment Log에 실패/성공이 분리 기록, 원인 명시
  4. 재시도로 실패 건을 독립적으로 복구 가능
  5. 원 실패 이력이 수정/삭제되지 않음 (append-only 원칙)
05

비고 (QA · 개발 메모)

운영 핵심: 반도체 현장에서 네트워크 이슈는 흔한 문제. 이 시나리오가 안정적이어야 제품이 실제 현장에서 쓸 수 있음.

Atomic Write: 장비 측에서 .psr 파일이 부분 기록 상태로 남으면 장비 오작동 위험. temp 파일 → rename 패턴 필수.

06

제안 사항

데이터 무결성 이 시나리오는 제품 품질의 최소 기준선입니다. Agent의 Atomic Write, 서버의 부분 실패 처리, 이력의 append-only가 모두 검증되어야 합니다.

QA 자동화 네트워크 단절 시뮬레이션이 포함된 시나리오로, 자동화 인프라에 네트워크 Fault Injection 기능이 필요합니다. 일반 UI 자동화로는 불가능하므로 별도 설계 필요.

UX 개선 Deployment Log 재시도 기능은 현장에서 가장 자주 쓰일 복구 도구입니다. Log 목록에서 원클릭 재시도가 가능해야 합니다.
