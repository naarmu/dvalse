URS_SET_ACC

계정 관리

User Requirement Security
01

기능 개요

RMS 사용자 계정을 생성·수정·비활성화하는 화면이다. 각 계정에는 고유 ID, 비밀번호, Role, 활성 상태가 부여되며, 사용자는 자신의 ID로 로그인하여 부여된 Role에 따른 권한으로 RMS를 사용한다.

Privilege가 Role 단위의 권한 정책을 정의한다면, Account는 그 Role을 실제 사람에게 매핑하는 화면이다. 두 화면이 함께 RMS의 사용자 보안 모델을 구성한다.

02

사용자 니즈

주요 사용자

  • RMS 관리자 — 신규 사용자 계정 발급 및 관리
  • Vendor — 시스템 전체 계정 관리·감사

Pain Point (RMS 도입 전)

  • 계정 발급·회수가 일관된 절차 없이 진행됨
  • 퇴사자·이동자 계정이 정리되지 않고 남는 경우가 빈번
  • 비밀번호 재설정 등 일상 운영 절차가 표준화되어 있지 않음

기대 효과

  • 계정 발급·회수의 표준화된 절차
  • 비활성화로 이력은 보존하면서 접근만 차단
  • 모든 계정 변경의 자동 이력 기록
03

기능 요구사항

■ FR-1. 계정 생성

  • FR-1.1 사용자는 신규 계정을 생성할 수 있어야 한다. 입력 필드는 ID(필수), 초기 비밀번호(필수), 표시 이름(선택), Role(필수), 부서(선택)이다.
  • FR-1.2 ID는 RMS 내에서 유일해야 한다. 중복 ID 입력 시 생성을 차단한다.
  • FR-1.3 ID 중복 검사는 대소문자를 구분하지 않는다.
  • FR-1.4 초기 비밀번호는 최소 길이·문자 종류 등의 정책을 충족해야 한다.
  • FR-1.5 사용자가 처음 로그인하면 비밀번호 변경을 요구하는 옵션을 제공할 수 있다.

■ FR-2. 계정 정보 변경

  • FR-2.1 사용자는 기존 계정의 Role, 표시 이름, 부서 등을 변경할 수 있어야 한다.
  • FR-2.2 변경 사항은 다음 로그인부터 적용된다.
  • FR-2.3 Role 변경은 즉시 권한에 반영되며 이력에 기록된다.

■ FR-3. 활성화 · 비활성화

  • FR-3.1 사용자는 계정을 비활성화(Disable)하여 로그인을 차단할 수 있어야 한다.
  • FR-3.2 비활성화된 계정은 로그인 시 별도 안내 메시지가 표시된다.
  • FR-3.3 비활성화된 계정은 다시 활성화(Enable)할 수 있다.
  • FR-3.4 비활성화 상태에서도 계정의 모든 이력은 보존된다.

■ FR-4. 비밀번호 재설정

  • FR-4.1 관리자는 사용자의 비밀번호를 재설정할 수 있어야 한다.
  • FR-4.2 재설정된 비밀번호는 평문으로 표시되지 않으며, 임시 비밀번호 발급 또는 사용자에게 직접 새 비밀번호를 설정하도록 하는 방식 중 하나를 사용한다.
  • FR-4.3 비밀번호 재설정 후 사용자는 다음 로그인 시 비밀번호를 변경하도록 강제할 수 있다.
  • FR-4.4 비밀번호 재설정 이력은 자동 기록된다.

■ FR-5. 계정 삭제

  • FR-5.1 계정의 영구 삭제(Hard Delete)는 지원하지 않는다. 비활성화로 대체한다.
  • FR-5.2 계정이 남긴 이력(작업 이력, 배포 이력 등)은 절대 삭제되지 않는다.

■ FR-6. Vendor 계정 보호

  • FR-6.1 마지막 남은 활성 Vendor 계정은 비활성화할 수 없다.
  • FR-6.2 마지막 남은 활성 Vendor 계정의 Role은 변경할 수 없다.

■ FR-7. 변경 이력

  • FR-7.1 모든 계정 변경(생성·Role 변경·비활성화·비밀번호 재설정)은 자동 기록된다.
  • FR-7.2 기록 항목에는 변경자, 변경 시각, 대상 계정, 변경 내용이 포함된다.
  • FR-7.3 변경 이력은 수정·삭제할 수 없다.
04

비기능 요구사항

■ NFR-1. 권한

  • NFR-1.1 Account 화면 접근은 Vendor, Admin만 가능하다.
  • NFR-1.2 Vendor 계정 생성·삭제는 Vendor만 수행할 수 있다.
  • NFR-1.3 Admin은 Engineer, Operator 계정만 관리할 수 있다.

■ NFR-2. 보안

  • NFR-2.1 비밀번호는 단방향 해시(예: bcrypt) + Salt로 저장된다.
  • NFR-2.2 비밀번호는 어떤 화면에서도 평문으로 표시되지 않는다.
  • NFR-2.3 모든 계정 관리 통신은 TLS 암호화 채널 위에서 수행된다.

■ NFR-3. 가용성

  • NFR-3.1 마지막 활성 Vendor 계정의 보호 메커니즘이 모든 경로(UI, API)에서 동작해야 한다.
05

제약사항 · 정책

  • 정책 계정의 영구 삭제(Hard Delete)는 지원하지 않는다. 비활성화(Disable)로 대체한다. 이력 무결성을 위함이다.
  • 정책 비활성화된 계정의 모든 이력(작업·배포·로그인 등)은 보존된다.
  • 정책 마지막 남은 활성 Vendor 계정은 비활성화하거나 Role을 변경할 수 없다. 시스템 잠금 방지가 목적이다.
  • 정책 ID 중복 검사는 대소문자를 구분하지 않는다.
  • 정책 비밀번호 재설정 후 사용자에게 첫 로그인 시 비밀번호 변경을 강제할 수 있다.
  • 정책 Admin은 Engineer, Operator 계정만 관리할 수 있다. Vendor 계정 관리는 Vendor만 가능하다.
  • 제약 비밀번호는 단방향 해시 + Salt로 저장되며 어떤 화면에서도 평문 노출되지 않는다.
  • 제약 모든 계정 변경 이력은 자동 기록되며 수정·삭제할 수 없다.

  • TC_SET_ACC_002 — 신규 사용자 계정 생성 (ID · PW · Role) (FR-1)
  • TC_SET_ACC_003 — 중복 ID 생성 차단 (FR-1.2, FR-1.3)
  • TC_SET_ACC_004 — 계정 Role 변경 (FR-2)
  • TC_SET_ACC_005 — 계정 비활성화 · 활성화 (FR-3)
  • TC_SET_ACC_006 — 비밀번호 재설정 (FR-4)
  • TC_SET_ACC_007 — 계정 삭제 및 제약 조건 (FR-5, FR-6, NFR-3.1)
  • TC_SET_ACC_008 — 계정 변경 이력 기록 (FR-7)
