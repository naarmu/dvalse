§0

설계 의도 세 문장으로 요약

Park.ds는 Park Systems의 전사 소프트웨어·웹·펌웨어 UI를 위한 단일 디자인 언어다. 1차 독자는 내부 개발진·디자이너이며, 이 사이트는 마케팅 쇼케이스가 아니라 디자이너와 개발자 간의 계약서이자 의사결정 지원 시스템이다. Spectrum의 구조를 참조하되, WPF/DevExpress 현실과 solo 운영 체제에 맞게 축소·재조합한다.

세 가지 핵심 원칙

P1
내부 전용 · 계약 언어

이 사이트는 교육 자료가 아니다. 철학적 서문은 최소화하고, 결정된 규칙·그 근거·개발 핸드오프 방법을 우선한다. 읽는 사람은 이미 맥락을 안다.

P2
플랫폼 중립 · 구현 명시

Foundation(토큰·규칙)은 플랫폼에 무관하게 선언한다. 각 구현 스택(WPF·Web·Firmware)의 handoff는 별도 레이어로 분리해 토큰 오염을 막는다.

P3
점진 성장 · 상태 명시

모든 문서는 READY · DRAFT · TEST · TODO 중 한 상태를 명시한다. v0.1은 Foundation 3개와 Icon 3개만 drafted 상태여도 괜찮다 — 진행 중임을 정직하게 드러낸다.

§1

Park.ds 사이트는 다음 7개 섹션으로 구성된다. 순서는 의존성 순이다 — Foundation이 Iconography를 전제하지 않고, Components는 Foundation+Icon 위에 세워진다.

왜 Patterns 섹션을 두는가? Park.ds의 실제 가치는 "Measurement View가 어떻게 생겨야 하는가" 같은 제품 수준 결정을 문서화할 때 나온다. Spectrum의 Patterns 섹션과 동일한 역할 — 여러 Component를 조합한 화면 레시피. Components 층이 비어 있어도, Patterns 층에는 "WLI의 표준 측정 뷰" 같은 실제 제품 페이지를 먼저 문서화할 수 있다.

§3

멀티플랫폼 확장 전략 WPF · Web · Firmware 삼축 대응

"전사 SW + 웹 + 펌웨어"라는 제품 범위는 Foundation 문서들의 기본 전제를 흔든다. 현재 Foundation은 FHD 24" WPF/DevExpress 데스크톱 환경에 최적화되어 있다. 이를 유지하면서 세 스택 모두를 커버하려면 Foundation을 2층으로 분리해야 한다.

2-Layer Foundation Platform-Neutral → Platform-Specific

Foundation Neutral Layer (권위) WPF / DevExpress Web Firmware
Color OKLCH 토큰 (22개 Semantic) ResourceDictionary · Theme Designer 매핑 ✓ CSS Custom Properties (변환 필요) RGB565/RGB888 테이블 (변환 필요)
Typography Role-based 4서체 · 7-layer scale Pancras*TextBlock Style ✓ CSS class + @font-face Pretendard 비지원 — 대체 서체 정의 필요
Spacing 4px hybrid scale · Density 2단계 XAML Spacing·Padding 값 ✓ CSS rem 기반 재매핑 픽셀 그리드 (해상도별 재계산)
Icon SVG 원본 · currentColor ResourceDictionary + PathGeometry ✓ SVG 그대로 ✓ 비트맵 스프라이트 변환 (화면 해상도별)
읽는 법. 녹색은 현재 문서화되어 있음, 주황은 변환 규칙 필요, 회색은 별도 전략 필요. 대부분이 주황이라는 것은 Foundation은 잘 짜였지만 Neutral Layer가 명시되어 있지 않다는 의미다. v0.2 작업의 가장 큰 주제가 여기다.

Neutral Layer 도입 제안 Design Tokens as Single Source of Truth

각 Foundation 문서의 상단에 JSON 또는 YAML 형식의 토큰 선언을 추가한다. 이 토큰이 Neutral Layer이며, 아래 세 스택의 구현은 모두 이 선언에서 파생된다. Style Dictionary(Amazon), Design Tokens W3C Spec, Tokens Studio 등의 도구를 써서 자동 변환 파이프라인을 구축한다.

Neutral Token (single source) tokens/color.json — OKLCH + hex 이중 표기 tokens/typography.json — size · weight · family tokens/spacing.json — 4px scale tokens/icons.json — 235 key → path 매핑 ↓ 변환 Stack-Specific Output build/wpf/App.xaml — ResourceDictionary build/web/tokens.css — CSS Custom Properties build/firmware/theme.h — C header with color table
이 구조의 큰 장점. 테드님이 Foundation 문서에서 하나의 토큰을 바꾸면, 세 스택의 빌드물이 자동으로 갱신된다. 지금처럼 "WPF 개발진에게 Typography XAML Update 문서를 따로 보내는" 일이 사라지고, 개발진은 최신 빌드를 가져가기만 하면 된다. 이게 디자인 시스템과 스타일 가이드의 결정적 차이다.

단점도 명확. 초기 파이프라인 구축 비용이 크다. v0.2에서 바로 도입할지, v0.3 이후 목표로 둘지는 Open Question에 포함한다.

§6

Open Questions 다음 결정이 필요한 지점

IA 수준의 미결정 사항

  1. Q1 · Components 전략. DevExpress 스타일 가이드만 vs DevExpress + Park 커스텀 하이브리드 vs 완전 커스텀 세트. 각 옵션의 엔지니어링 비용·기간 견적이 필요. 제안: 하이브리드. 90%는 DevExpress를 Foundation 토큰으로 스타일, 10%는 Park 고유 UX가 필요한 경우(예: Equipment State indicator)만 커스텀. 결정 주체는 개발 리드와 공동.
  2. Q2 · 제품 범위의 현실 확인. "전사 SW + 웹 + 펌웨어"가 실제 Park Systems의 내부 현황에 비춰 1~2년 내 목표인가, 3~5년 비전인가. v0.2 scope에 영향. 제안: 1~2년 내 목표는 "전사 WPF SW"로 좁히고, 웹/펌웨어는 Neutral Layer 도입으로 "준비만" 해두는 방향. 실제 변환 파이프라인은 실제 제품이 생길 때.
  3. Q3 · Neutral Layer 도입 시점. v0.2에서 Style Dictionary 기반 토큰 파이프라인을 구축할지, v0.3 이후로 미룰지. 구축 비용이 크고 solo 운영에선 큰 투자. 제안: v0.3 이후. 현재는 수작업 handoff가 유지 가능한 규모이며, 컴포넌트 수가 늘어나기 시작할 때가 적기.
  4. Q4 · 물리 폴더 구조 재편 시점. 현재 park.ds/ 폴더에 모든 문서가 평면 배치되어 있음. foundations/, iconography/처럼 섹션별 폴더로 재편할지, 유지할지. 제안: v0.2 작업 시작 시 재편. 새 문서가 추가되기 전에 정리가 쉬움. GitHub 커밋 한 번으로 이동.
  5. Q5 · 문서 버전 정책. 파일명에 v0.1을 박을지(현재 방식), 파일명은 고정하고 문서 상단 메타에만 버전을 표시할지. 제안: 파일명에서 버전 제거. Git이 버전을 관리하게 하고, 파일명은 안정적으로 유지 (링크 깨짐 방지).
  6. Q6 · Patterns 섹션 착수 시점. Components 이전에 Patterns부터 써도 되는가? 실제 제품 화면을 먼저 문서화하는 bottom-up 방식이 의미가 있을지. 제안: 의미 있음. "WLI 측정 뷰 하나"를 먼저 Patterns에 써보면, 어떤 Components가 실제로 필요한지 역으로 드러난다. v0.2에서 Pattern 1개를 시험 삼아 써보는 걸 권장.
