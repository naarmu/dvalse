§1 SVG Pipeline 정책 85개로 확장하기 전 확정 필요

9개 테스트에서 나온 문제를 85개로 곱하면 수백 곳을 고쳐야 합니다. Park.ds가 Figma export를 어떻게 처리할지 한 번만 정해두면 나머지는 기계적입니다.

후처리 파이프라인 (제안)

  1. 원본 보존 — Figma export 그대로 icons/raw/{FigmaName}.svg에 저장 (현재 상태)
  2. 1차 변환 — fill="black" → fill="currentColor", <svg>의 fill="none"은 제거, 불필요 속성(xmlns 제외) 정리
  3. 리네이밍 — 06 Icon Naming 규칙에 따라 icons/{tier}/{Namespace.}{Name}.svg로 이동·개명
  4. 최적화 — SVGO로 path 좌표 소수점 3자리 제한, 중복 속성 제거 (파일크기 평균 40% 감소 기대)
  5. 개발 대상 배포 — WPF 개발진용 ResourceDictionary 자동 생성 (v1 publish 시점)
제가 자동화 가능한 범위: 1~3번은 업로드된 SVG에 대해 즉시 스크립트로 처리 가능. 4번 SVGO는 제 환경에서 실행 불가 — 추후 박사님 로컬 또는 CI에서. 5번은 XAML 문법이라 별도 작업.

§3 박사님 결정 필요 85개 확장 전 확인

질문

  1. 후처리 스크립트 실행 여부. 제가 icons/raw/*.svg를 읽고 fill="currentColor"로 자동 변환 + 규칙에 따라 icons/common/*.svg로 복사해도 될까요? (원본은 보존)
  2. Setting vs Settings. DevExpress/WPF 리소스에서 이미 Setting으로 쓰고 있나요? 쓰고 있다면 Setting 유지 · alias 없이 규칙에서 제외. 아직 안 쓰였다면 Settings로 교정.
  3. Configure 아이콘의 의미. "톱니 12개 고리" 디자인이 Setting과 의미 중복 아닌가요? Setting = 일반 설정, Configure = 장비 configure 같은 구분이 있으면 공유해주세요.
  4. Palette 아이콘의 분류. 팔레트는 공통 UI에서 거의 안 쓰이는 편인데, WLI의 Color/LUT 선택처럼 specific으로 옮기는 게 맞는지? 아니면 common 유지?
  5. 전체 85개 SVG 업로드 타이밍. 이 테스트 OK면 나머지 76개도 같은 방식으로 zip이 아닌 섹션별 드래그앤드롭으로 올려주실 수 있을까요?
