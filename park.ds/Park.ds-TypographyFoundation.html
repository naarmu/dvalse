§0

설계 의도

Park.ds의 활자는 Pretendard 한 벌을 중심에 두고, 역할이 명확한 경우에만 보조 서체를 허용한다. 수치는 두 층위로 나뉜다 — 사용자가 입력·편집하는 값은 JetBrains Mono, 장비가 방출하는 고유값(voltage, framerate 등)은 DSEG14로 표기하여 "내가 만든 값 / 기계가 말하는 값"을 시각적으로 구분한다. 보고서 출력(Report Export)에만 한정하여 Source Serif 4를 허용하며, 화면 UI에는 세리프를 쓰지 않는다.

§3

Weight Ladder 4단계 · 그 이상은 금지

Pretendard는 100~900까지 제공하지만 Park.ds는 4단계만 사용한다. 위계는 크기로 세우고, 굵기는 "강조해야 할 때만" 쓴다. 특히 Thin(100) / ExtraLight(200) / Light(300)은 장비 운용 환경(FHD 24")에서 가독성을 해치므로 전면 금지한다.

측정 시작
400 · Regular · Body · Annotation
측정 시작
500 · Medium · Label · 강조
측정 시작
600 · SemiBold · Headline · Title
측정 시작
700 · Bold · Digital(DSEG) 전용

* 100/200/300(Thin~Light), 800(ExtraBold), 900(Black)은 사용 금지. 산업용 톤과 충돌.
* Label은 Uppercase 강제 금지 — 한/영 혼용에서 어색함. 강조는 위치·색으로 준다.
* Italic은 로그·인용·플레이스홀더에만 사용.

§4

Numeric Rules tabular figures + 서체 분리

숫자는 흔들리지 않아야 한다. Pretendard 본문에도 font-feature-settings: "tnum"을 기본 적용해 숫자가 동일 폭을 갖도록 한다. 이로써 측정값이 업데이트될 때 숫자 레이아웃이 떨리지 않는다.

  999,999
With tnum (tabular) · Park.ds 기본값

2,222,222

  999,999

* tnum은 디자인 품질의 기본. 그러나 일부 UI(예: 좁은 공간의 이름 + 숫자 혼용)에서 과도한 공백이 생기면 해당 지점만 font-variant-numeric: normal로 끌 수 있다.

§6

피드백 요청

이 초안에서 결정이 필요한 지점

  1. Weight Ladder — 현재 4단계(400/500/600/700). 실제 제품에서 더 촘촘한 위계가 필요할지, 아니면 이대로 충분한지.
  2. Display 계열 크기 — DSEG 대형 수치 표시 기본값을 44px로 두었다. 실제 장비 모니터링 뷰에서 적절한지, 더 크거나 작아야 할지.
  3. Italic 허용 범위 — 로그·플레이스홀더·인용에만 허용으로 제한했다. Report Export에서도 이 규칙을 유지할지.
  4. Report Export 서체 — Source Serif 4로 잡았는데, 실제 고객 리포트 관행(예: Times·Arial 같은 관습)과 충돌이 있다면 알려주시길.
  5. UI 밀도 — 13px 기본으로 한 미리보기(§5)의 밀도·가독성이 현장 환경에 적합한지.
