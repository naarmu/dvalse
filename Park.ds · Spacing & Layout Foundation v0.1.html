§ 0

Philosophy — 왜 공간을 이렇게 설계했는가

notes

측정·분석·관리 SW의 화면은 정보 밀도가 높다. 한 화면에 수십 개의 값, 설정, 상태가 동시에 놓이며, 엔지니어는 이 화면을 8시간 이상 응시한다. 따라서 Park.ds의 공간 철학은 "넓게 쉬어주는 화면"이 아니라, 밀도를 해치지 않으면서 구조를 읽히게 하는 공간을 만드는 것이다.

· 공간은 장식이 아니라 구조이다. 여백은 요소를 "예쁘게 띄우는" 도구가 아니라 그룹을 정의하고 위계를 만드는 신호다.
· 4px 기반의 예측 가능한 격자를 쓴다. 모든 여백은 4의 배수 또는 허용된 2px 단계로 정렬되며, "대충 6px, 대충 10px"은 존재하지 않는다.
· 밀도를 두려워하지 않는다. Compact가 기본이다. 장비 앞 작업자를 위해 Comfortable 레벨을 옵션으로만 제공한다.

참고 — Typography가 13px body.sm을 기본으로 삼았으므로, 이 문서의 모든 높이·padding 값은 13px 본문 기준으로 맞춰져 있다.

§ 1

Base Unit & Scale — 4px 주축 + 2px 허용 hybrid

§1

기본 격자는 4px, 보조 격자는 2px이다. 4px은 모든 컴포넌트의 외곽 배치(margin, gap, container padding)에 쓰이고, 2px는 컴포넌트 내부(아이콘↔라벨, 버튼 내부 등 시각적 접합부)에서만 허용한다.

4px 8px 12px 16px 20px 24px 32px 40px
4px grid · strong = 8px

* 위 격자는 4px 단위이며, 진한 세로선은 8px 단위다. 모든 UI 요소는 시작 x좌표와 끝 x좌표가 진한 선(8px) 위에 놓이는 것을 원칙으로 하고, 내부 디테일에서만 연한 선(4px·예외 2px)을 허용한다.

Token Value Type Scale Usage

space.6 6 px ext 인풋 내부 수직 padding(13px 본문 기준 · Compact)
space.8 8 px core 폼 필드 사이 · 툴바 내부 · 인라인 카드 gap
space.10 10 px ext 표 셀 수평 padding · 버튼 내부 수평(sm)
space.12 12 px core 카드 내부 수직 · 작은 섹션 gap · 다이얼로그 버튼 간격
space.16 16 px core 패널/카드 기본 padding · 그룹 사이

space.40 40 px core 대형 섹션 사이 · 히어로 영역

* space.0은 토큰이지만 실제로는 여백 없음을 명시적으로 선언할 때만 쓴다.
* space.2px, space.6px는 Extended 스케일로, 컴포넌트 내부에서만 허용. 레이아웃(카드 간격, 섹션 gap 등)에는 절대 쓰지 않는다.
* 56px 이상은 토큰을 만들지 않고 multiple of 8 (64, 80, 96, …)을 쓴다 — 대형 여백은 의도된 예외이기 때문에 토큰으로 정당화하지 않는다.

§ 3

Inside vs Outside — padding과 margin을 섞지 않는다

§3

공간은 두 개의 서로 다른 책임으로 나뉜다. Inside(padding)는 컴포넌트 스스로가 소유하고, Outside(gap/margin)는 부모 레이아웃이 결정한다. 이 둘을 섞으면 동일한 컴포넌트가 위치에 따라 다른 여백을 갖는 사고가 발생한다.

§3.1 · Inside (padding)

컴포넌트가 스스로 가진다

버튼 내부 padding, 카드 내부 padding, 인풋 내부 padding 등. 컴포넌트 자체의 "기본 형태"에 포함되므로 외부에서 override하지 않는다.

Button.padX · Card.pad · Input.padX · Dialog.pad · Panel.pad
§3.2 · Outside (gap / margin)

부모 레이아웃이 결정한다

컴포넌트 간 간격은 부모의 gap으로만 준다. 자식이 스스로 margin을 갖지 않는다(= margin: 0이 기본). 레이아웃은 재배치에 자유로워야 한다.

Form.gap · Toolbar.gap · Stack.gap · Grid.gap
✓ DO — 부모가 gap, 자식은 padding만
<StackPanel Spacing="8">
  <Button Padding="12,0" />
  <Button Padding="12,0" />
</StackPanel>
✗ DON'T — 자식이 margin
<StackPanel>
  <Button Margin="0,0,8,0" />
  <Button Margin="0,0,8,0" />
</StackPanel>

§ 6

Do / Don't — 공간 설계 체크리스트

§6
✓ DO · 4의 배수
모든 여백은 토큰에서 고른다

margin: 8px; gap: 12px; padding: 16px 20px; — 토큰 스케일에 있는 값만 쓴다.

✗ DON'T · 임의값
"대충 10, 15, 18" 금지

margin: 10px; padding: 15px 18px; — 토큰에 없는 값은 리뷰에서 반려한다.

✓ DO · 부모가 gap
자식은 margin 0

StackPanel.Spacing · Grid gap · Flex gap 으로만 간격을 준다.

✗ DON'T · 자식이 margin
"마지막만 0" 패턴 금지

:last-child margin-right:0 같은 hack은 레이아웃 재배치 시 버그를 낳는다.

✓ DO · 비대칭 허용
수직 < 수평

Card padding 16/20, Dialog 20/24 — 읽는 축(수평)에 더 큰 여백을 준다.

✗ DON'T · 무분별한 대형 여백
64px 이상 금지

측정 SW 화면에서 40px를 넘는 빈 공간은 "정보 낭비"다. 필요하면 근거를 문서화.

§ 7

Open Questions — 개발진 협의 필요

§7

확정 필요

  1. Density 전환 방식 — 리소스 딕셔너리 교체(재시작 필요) vs DynamicResource(실시간). DevExpress Theme와 호환되는 방식으로.
  2. DevExpress GridControl 기본 RowHeight — 현재 여러 값이 혼재. RowAutoHeight=false · RowHeight=24로 통일?
  3. Inspector 접힘 동작 — 완전 숨김 vs 32px 아이콘 레일 유지 중 어느 쪽을 기본으로?
  4. MDI / Dockable 윈도우 — DockLayoutManager 사용 제품에서 Panel padding이 기본 3px로 들어가는데, 이를 재정의할지 DevExpress 규칙을 따를지.
  5. Comfortable의 실제 수요 — 현업에서 장갑 상태 입력이 있는 제품이 어느 것인지 재확인 (WLI에 해당?).
